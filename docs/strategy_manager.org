* Purpose

This document describes the proposed Strategy Manager architecture for
the Trader system.

The goal is to evolve live trading from:

#+begin_src text
one manually configured live strategy per instrument
#+end_src

toward:

#+begin_src text
one trader process owns one account
one account manages multiple <instrument, strategy> tuples
one instrument/account has at most one active strategy
regime and opportunity state determine which strategy should run
risk management controls whether orders are allowed and how large they are
#+end_src

This is not intended to replace the current strategy, backtest, or
live runner architecture. It adds an orchestration layer above
existing strategy execution.

* Current Relevant Architecture

The current system already has several useful building blocks.

** LiveStrategy

The root =trader= package defines =LiveStrategy=:

#+begin_src go
type LiveStrategy interface {
    Name() string
    Tick(ctx context.Context, price LivePrice, openTrades []LiveTrade) *LivePlan
}
#+end_src

This is the live trading interface. A strategy receives the current price and open trades for its instrument, then returns a =LivePlan=.

** Live Runner

The service layer has =RunLiveStrategy=, which runs one live strategy loop.

The current runner:

#+begin_src text
1. fetches current broker price
2. fetches open broker trades
3. converts broker state into LivePrice and LiveTrade values
4. calls strategy.Tick
5. executes closes first
6. opens at most one new position
7. records broker trade ownership for journaling
#+end_src

This is a good low-level runner, but it is not yet the Strategy Manager.

** Candle Strategy Adapter

The current =CandleStrategyAdapter= wraps backtest-style candle
strategies as live strategies.

It already:

#+begin_src text
- fetches completed OANDA candles
- warms strategy indicators
- feeds closed candles to the strategy
- advances regime and exit filters
- blocks opens that fail the regime filter
- converts StrategyPlan into LivePlan
#+end_src

This is important because live candle strategies can reuse existing
backtest strategy logic.

** Existing Gap

The current system can run strategies live, but it does not yet have a
full account-level orchestration layer that decides:

#+begin_src text
- which instruments should be watched
- which instruments are near a tradeable setup
- which strategy should own a specific instrument slot
- whether multiple strategies conflict
- whether account-level risk permits a new trade
- whether one trader process has exclusive ownership of the account
#+end_src

The proposed Strategy Manager fills this gap.

* Target Architecture

** High-Level Flow

#+begin_src text
Market Data / Candle Store / Broker Prices
        |
        v
Review Layer / Regime Classifier
        |
        v
Watchlist Manager
        |
        v
Opportunity Detector
        |
        v
Strategy Manager
        |
        v
Risk Manager / Position Sizer
        |
        v
Strategy Runner
        |
        v
Signal / Order Manager
        |
        v
Broker Adapter
#+end_src

** Operating Rule

#+begin_src text
One running trader process owns exactly one account.

One account may have multiple active <instrument, strategy> tuples.

For each <account, instrument> slot:
- zero or one active strategy
- zero or more observer strategies
- one active position owner
- risk manager can override everything
#+end_src

** Why One Active Strategy Per Instrument?

Multiple active strategies on the same instrument/account create
unclear ownership.

Bad live-trading state:

#+begin_src text
EUR/USD + EMA Pullback Strategy -> wants long
EUR/USD + Breakout Strategy     -> wants long
EUR/USD + Mean Reversion        -> wants short
EUR/USD + Exit Strategy A       -> wants exit
EUR/USD + Exit Strategy B       -> wants hold
#+end_src

Clean live-trading state:

#+begin_src text
Account: OANDA-Demo

Active:
- EUR/USD -> EMA Pullback Strategy
- GBP/USD -> Donchian Breakout Strategy
- USD/JPY -> Trend Continuation Strategy
- AUD/USD -> Bollinger Fade Strategy
#+end_src

Other strategies can still run in =observer= or =paper= mode, but only
one strategy should be allowed to place live orders for a given
account/instrument.

* Core Concepts

** Account Owner

A =TraderProcess= owns exactly one account.

#+begin_src text
trader-demo-oanda -> OANDA demo account
trader-live-oanda -> OANDA live account
trader-ibkr-paper -> IBKR paper account
trader-ibkr-live  -> IBKR live account
#+end_src

The system should prevent two live trader processes from owning the same account.

Recommended rule:

#+begin_src text
account_id -> exactly one active trader owner
#+end_src

** Instrument Slot

An =InstrumentSlot= represents one instrument in one account.

Example:

#+begin_src text
<OANDA-Demo, EUR_USD>
<OANDA-Demo, USD_JPY>
<OANDA-Live, EUR_USD>
#+end_src

The Strategy Manager should manage state per instrument slot.

** Instrument Slot State

Suggested lifecycle:

#+begin_src text
Inactive
  -> Watching
  -> NearSetup
  -> TradeCandidate
  -> StrategyActive
  -> PositionOpen
  -> Cooldown
  -> Invalidated
#+end_src

State descriptions:

| State            | Meaning                                               |
|------------------+-------------------------------------------------------|
| =Inactive=       | Instrument is not currently interesting               |
| =Watching=       | Higher timeframe regime says it is worth watching     |
| =NearSetup=      | Price is approaching a tradeable setup area           |
| =TradeCandidate= | Setup confirmed enough to select a strategy           |
| =StrategyActive= | A strategy has been assigned and is receiving candles |
| =PositionOpen=   | Broker has an open position owned by the strategy     |
| =Cooldown=       | Recently exited; wait before re-entry                 |
| =Invalidated=    | Regime/setup broke; remove from active consideration  |

** Strategy Assignment

A strategy assignment binds an instrument slot to one active strategy.

#+begin_src text
account_id
instrument
strategy_name
strategy_instance_id
mode
assigned_at
assignment_reason
#+end_src

Modes:

#+begin_src text
disabled
watch_only
paper_only
active
#+end_src

Only one strategy can be =active= per =<account, instrument>=.

** Position Ownership

Each broker position should have exactly one owning strategy instance.

#+begin_src text
account_id
instrument
broker_trade_id
strategy_name
strategy_instance_id
entry_signal_id
risk_model_id
opened_at
#+end_src

Only the owning strategy should emit strategy-based exits.

The Risk Manager may still force exits globally.

* Review Layer / Regime Classifier

** Responsibility

The Review Layer identifies the strategic regime for each instrument and timeframe.

Examples:

#+begin_src text
EUR/USD D1 = bullish trend
GBP/USD H4 = range
USD/JPY D1 = bullish trend but intervention risk
AUD/USD H1 = compression
#+end_src

** Daily or Realtime?

Use both, but for different purposes.

#+begin_src text
D1 / H4 review:
  strategic regime and trade context

H1 / M15 / realtime candles:
  tactical confirmation, invalidation, and opportunity state
#+end_src

The daily review should answer:

#+begin_src text
- Is this instrument worth watching today?
- What is the higher-timeframe regime?
- What direction is favored?
- What strategy families are eligible?
- What support/resistance zones matter?
#+end_src

The realtime layer should answer:

#+begin_src text
- Has the setup confirmed?
- Has the regime degraded?
- Has price entered a no-trade zone?
- Has spread or volatility become unacceptable?
- Should the instrument move from Watching to TradeCandidate?
#+end_src

** Proposed Types

#+begin_src go
type Regime string

const (
    RegimeUnknown          Regime = "unknown"
    RegimeTrendingBullish  Regime = "trending_bullish"
    RegimeTrendingBearish  Regime = "trending_bearish"
    RegimeRanging          Regime = "ranging"
    RegimeCompression      Regime = "compression"
    RegimeVolatile         Regime = "volatile"
    RegimeNoTrade          Regime = "no_trade"
)

type RegimeSnapshot struct {
    Instrument  string
    Timeframe   string
    Regime      Regime
    Confidence  float64
    Direction   string
    UpdatedAt   time.Time
    Reasons     []string
}
#+end_src

** Implementation Notes

The current =RegimeFilter= is strategy/backtest oriented. It is good for per-strategy safety, but the Strategy Manager also needs account-level regime snapshots.

Recommended distinction:

#+begin_src text
RegimeFilter:
  strategy-local guard used during backtests and live candle execution

RegimeClassifier / ReviewLayer:
  system-level service that classifies instruments and updates watchlist state
#+end_src

* Watchlist Manager

** Responsibility

The Watchlist Manager tracks which instruments deserve attention.

It does not place trades.

It should promote/demote instruments based on review and realtime state.

#+begin_src text
Review says:
  EUR/USD is bullish trending and worth watching

Watchlist says:
  EUR/USD is Watching

Realtime says:
  EUR/USD is pulling back toward EMA20

Watchlist says:
  EUR/USD is NearSetup
#+end_src

** Proposed Type

const (

* Opportunity Detector

** Responsibility

The Opportunity Detector decides when a watched instrument becomes actionable.

The Watchlist Manager says:

#+begin_src text
EUR/USD is interesting
#+end_src

The Opportunity Detector says:

#+begin_src text
EUR/USD has confirmed a tradeable setup
#+end_src

Examples:

#+begin_src text
- H1 candle closes above EMA20 in bullish trend
- price breaks Donchian channel in trending regime
- price reaches range support in ranging regime
- compression breakout confirms with close and volatility expansion
#+end_src

** Why Separate It From Strategy Manager?

The Strategy Manager should select and run strategies.

The Opportunity Detector should decide whether an instrument has become actionable enough to assign a strategy.

This keeps responsibilities clean.

** Proposed Type

#+begin_src go
type Opportunity struct {
    AccountID     string
    Instrument    string
    Timeframe     string
    Side          string
    Regime        RegimeSnapshot
    StrategyFamily string
    CandidateStrategies []string
    Confidence    float64
    Reason        string
    DetectedAt    time.Time
}
#+end_src

** New Code Needed

#+begin_src text

service/opportunity_detector_test.go
#+end_src

For v1, this can be rule-based and simple.

* Strategy Manager

** Responsibility

The Strategy Manager assigns one active strategy to one instrument slot.

It should answer:

#+begin_src text
Given this account, instrument, regime, and opportunity,
what single strategy should be active?
#+end_src

It should not perform broker execution directly.

** Responsibilities

#+begin_src text
- maintain instrument slot state
- select one active strategy per <account, instrument>
- start/stop strategy runners
- keep strategy assignment records
- prevent multiple active strategies on same slot
- allow observer/paper strategies separately
- consult risk manager before allowing active order flow
- expose status to CLI/REST/MCP/UI
#+end_src

* Risk Manager

** Current State

Live orders already support risk percentage, max units, and max position USD through =LiveOpenRequest= and =LiveRunConfig=.

That is useful, but it is local to the live runner.

The Strategy Manager needs an account-level Risk Manager that can decide whether a new trade is allowed before the order request reaches the broker.

** Responsibilities

#+begin_src text
- account-level max open trades
- max risk per trade
- max total open risk
- max risk per instrument
- max risk per strategy family
- max correlated exposure
- max drawdown circuit breaker
- max position notional
- spread/volatility guards
- confidence-adjusted sizing
#+end_src

* Live Runner Changes

** Current Runner

=RunLiveStrategy= currently runs one strategy for one instrument and directly executes the returned =LivePlan=.

** Needed Change

Keep =RunLiveStrategy= as the low-level runner, but add a higher-level managed runner that wraps or coordinates it.

** Option A: Keep Existing Runner and Add Strategy Manager Above It

#+begin_src text
Strategy Manager starts/stops RunLiveStrategy goroutines.
Each active <instrument, strategy> tuple gets one runner.
#+end_src

Pros:

#+begin_src text
- minimal rewrite
- reuses existing tested live execution path
- easy v1
#+end_src

Cons:

#+begin_src text
- account-level risk checks may be duplicated or awkward
- harder to coordinate cross-instrument exposure before strategy.Tick
#+end_src

** Option B: Create Account-Level Live Orchestrator

#+begin_src text
One account-level loop owns broker polling, open trade state, watchlist state, risk checks, and strategy assignment.
#+end_src

Pros:

#+begin_src text
- cleaner long-term architecture
- easier account-level risk
- easier one-account ownership
- easier UI status
#+end_src

Cons:

#+begin_src text
- more new code
- more careful concurrency design
#+end_src

** Recommendation

Use Option A for v1 and evolve toward Option B.

V1:

#+begin_src text
Strategy Manager starts/stops existing RunLiveStrategy loops.
Risk checks are inserted before order placement.
#+end_src

V2:

#+begin_src text
Account-level LiveOrchestrator owns all instrument slots and order flow.
#+end_src

** Code to Modify

#+begin_src text

live_strategy.go
service/candle_strategy_adapter.go
#+end_src

Specific changes:

#+begin_src text

- add order rejection logging with structured reasons
- record strategy ownership for every opened broker trade
#+end_src

* Account Ownership Lock

** Why Needed

Only one running trader should own a live account.

Without an ownership lock, two processes could accidentally trade the same account.

** V1 Implementation

Use a local filesystem lock for desktop/local operation.

#+begin_src text
~/.local/state/trader/account-locks/<account-id>.lock
#+end_src

The process should:

#+begin_src text
1. acquire lock on startup
2. write process id, hostname, account id, started_at
3. refresh heartbeat periodically
4. refuse to trade if lock is held by another live process
5. release on clean shutdown
#+end_src

** V2 Implementation

Use SQLite or another durable lease store.

#+begin_src text
account_leases
--------------
account_id
owner_id
hostname
pid
started_at
heartbeat_at
expires_at
#+end_src

** New Code Needed

#+begin_src text

service/account_lock_test.go
#+end_src

* Design Decisions To Preserve

#+begin_src text
- one trader process owns one account
- one account can manage many instruments
- one account/instrument has at most one active strategy
- observer strategies may exist but cannot place live orders
- every live position has one owning strategy instance
- risk manager can override strategy decisions
- strategy-level regime checks remain as an insurance policy
- daily/H4 review provides strategic context
- realtime candles provide tactical confirmation and invalidation
#+end_src

* Final Recommendation

Do not rewrite the existing live runner first.

The current live runner and candle adapter already provide a useful execution path. Build the Strategy Manager as an orchestration layer above them.

Recommended build sequence:

#+begin_src text
1. Add account ownership lock.
2. Add account-level risk manager.
3. Add strategy metadata/descriptors.
4. Add strategy manager state and assignment.
5. Use the existing live runner as the execution worker.
6. Add watchlist and opportunity detection.
7. Later evolve into a full account-level live orchestrator.
#+end_src

This gives the system better safety and control quickly while preserving existing backtest/live strategy reuse.
