This doc describes the target shape so it survives the refactor and can
fold directly into the manual's Architecture section once implemented —
see =docs/Plans/service-refactor.org= for the related =service.Service=
god-object refactor this unblocks.

* Ownership hierarchy

A =Broker= is an execution venue. A =Broker= can have more than one
=Account=. An =Account= holds =Orders=, =Positions=, and =Trades=. This is
a strict three-level ownership chain:

#+begin_example
Broker
  └─ Account
       └─ Order / Position / Trade
#+end_example

Market data is explicitly *not* part of this chain. Historical candle data
and live price/tick streaming are independent of any account — some data
providers (Dukascopy) aren't brokers at all, and even for a vendor that is
both (OANDA), live streaming is conceptually =DataProvider='s
responsibility, not =Broker='s, even though the bytes happen to arrive over
the same connection. See =docs/Manual/DataManager.org= for the data side;
this doc only covers execution.

* Package layout

Flat at repo root — no =providers/= or =internal/= umbrella (considered and
rejected; see =docs/Plans/service-refactor.org= for why). =brokers/= and
=account/= already exist at root today and stay there.

#+begin_example
brokers/
  broker.go              -- Broker interface: PlaceOrder, CloseTrade,
                             GetAccountSummary, StreamFills, ...
  oanda/                  -- concrete Broker implementation (exists today)
  sim/                    -- concrete Broker implementation (exists today)
  alpaca/                 -- future
  interactivebrokers/     -- future

account/
  account.go               -- Account (domain type: an account at a broker)
  ledger.go                -- Ledger (renamed from account.Broker — see below)
  order.go                 -- Order (new: canonical, broker-agnostic)
  trade.go, trade_position.go, trade_lot.go, fill.go  -- existing, unchanged
#+end_example

** The =Broker= interface

Doesn't exist yet. Lives in =brokers/broker.go=, satisfied by =oanda.Client=
and =sim.Simulator= (and, eventually, =alpaca=/=interactivebrokers=
implementations). Only covers execution — placing/closing orders, reading
account state, streaming fills/transactions. Deliberately excludes
candle/price history: a broker that also happens to serve market data
(OANDA) satisfies =DataProvider= as a *separate* interface, not as part of
=Broker=. Resist adding data-fetching methods to =Broker= just because one
concrete implementation happens to support both.

** Renaming =account.Broker= → =account.Ledger=

=account/broker.go= currently defines a =Broker= struct — but it's the
in-memory bookkeeping ledger (open lots, balance tracking), used by both
backtest (=account.NewBroker("sim")=) and, implicitly, live position
tracking. That's a different concept from the =brokers.Broker= interface
above (an execution venue), and having both named "Broker" one package
apart (=account.Broker= vs. =brokers.Broker=) is a landmine for anyone
reading imports. Renaming the ledger to =account.Ledger= frees "Broker" to
mean only the execution-interface concept, consistently, everywhere.

** New: =account.Order=

No broker-agnostic =Order= type exists today. Live code
(=service/orders.go=) talks directly in OANDA's vocabulary —
=oanda.CloseTradeResult=, =oanda.OpenTrade= — which means every additional
broker would require its own request/result shapes threaded through every
service method that touches orders. =account.Order= becomes the canonical
shape; each =brokers/<name>= package is responsible for translating
=account.Order= to/from its own wire format on the way in, and translating
fills back into =account.Fill=/=account.Trade= on the way out.

=Order= lives in =account/=, not its own top-level package: an order
belongs to an account tightly enough that a fourth root package felt like
more ceremony than the size warrants.

* Why broker/account first, not the Service split

=service/account_*.go='s =Account= (a live per-session cache/wrapper —
distinct from =account.Account=, the domain type above; see the naming
note below) currently holds a raw =*oanda.Client= via a back-reference to
=*Service=. The Service/=Runnable= split (=docs/Plans/service-refactor.org=)
can't be designed properly without first knowing what a decoupled
=service.Account= should depend on instead — and that answer is
"=brokers.Broker=," which doesn't exist until this refactor lands.
Sequencing broker/account first means the Service split is informed by a
real interface instead of a guess, and de-risks the larger, messier
refactor by landing the smaller, self-contained one first.

** Naming note: three things currently called "Account"

Worth being explicit about, since it's a live source of confusion during
this refactor:

1. =account.Account= (root =account/= package) — the domain bookkeeping
   type: balance, open lots, PNL. Broker-agnostic.
2. =service.Account= (=service/account_session.go=) — a live per-request
   session cache/wrapper: holds an ID, a back-reference to =*Service=, and
   an =AccountSnapshot=. This is the thing the god-object refactor is
   about; unrelated to (1) except by name.
3. =oanda.AccountSummary= / =oanda.AccountDetails= (=brokers/oanda=) — wire
   DTOs from OANDA's API.

(3) stays a wire DTO, untouched. (1) and (2) are **merging** — see Phase 3
below — resolving the ambiguity by deciding there should only be one
=Account= type, not by permanently keeping two.

* Phase 3: merge =service.Account= into =account.Account=

Decided 2026-07-21, not yet implemented. Rather than keep the domain ledger
(=account.Account=) and the live session wrapper (=service.Account=) as two
separate types that happen to share a name, they become one
=account.Account=, and the session-registry constructors currently on
=service.Service= (=Account=, =Accounts=, =FirstAccount=, =DefaultAccount=)
move into =account/= with it. =service/orders.go=' order-placement logic
(=PlaceMarketOrder=, =CloseTrade=, =UpdateTradeStop=, =ListOpenTrades=) is
expected to move along with =Account= at the same time, per the earlier
discussion — its methods are all on =*Account= already, so leaving them
behind in =service/= once =Account= moves would split one type's behavior
across two packages for no reason.

** Known collision to resolve: =Account.ID=

=account.Account= already has an =ID= field
(=account/account.go=, set via =idgen.NewULID()= in =NewAccount=) — but
it's an internally-generated ULID, purely a ledger-instance identifier.
Grepped: it is never read outside =account/= itself, and every non-test
reference is a hardcoded literal (=ID: "test"=) — it is not load-bearing in
backtest or engine today.

=service.Account.ID=, meanwhile, *is* load-bearing — it's the real OANDA
account identifier string (e.g. =101-004-12345-001=) threaded through
every live broker call (=a.broker().GetOpenTrades(ctx, a.ID)= and
similar).

These are different concepts sharing a field name. Resolution: repurpose
=Account.ID= to mean the live/broker account identifier (matching
=service.Account='s usage, since that's the one anything actually depends
on); backtest's =NewAccount(name, deposit)= either stops populating =ID=
with a ULID or keeps doing so harmlessly — confirmed safe either way, since
nothing reads it in that path.

** What the merged =account.Account= needs

Beyond the existing ledger fields (=Balance=, =Equity=, =Lots=, =Trades=,
=RiskFraction=, ...), the live-session half needs:

- *No =Broker= field.* Corrected 2026-07-22: ownership points the other
  way — "a Broker can have more than one Account" (this doc's own opening
  hierarchy), and every =brokers.Broker= method already takes =accountID=
  as a parameter, confirming the interface itself is shaped as "one broker
  addressing many accounts," not "one account holding one broker." What
  submits an order needs a =Broker= handle to call, but that handle isn't
  something =Account= owns — see Phase 4 for where it actually lives.
- =Log *slog.Logger= — replaces =a.svc.Log=.
- The =AccountSnapshot= cache (=snapMu=/=snapshot=) — moves over unchanged;
  it already has zero back-reference to =*Service= (see
  =docs/Plans/service-refactor.org='s Phase 1 note — this was already
  identified as the one piece with no cycle risk). Its role narrows once
  Phase 4's event-driven fills land — see Phase 4's source-of-truth note.
- Pricing/streaming (=GetPricing=, =StreamPricing=, used by
  =prices.go=/=live_runner.go=/=pip_values.go=) stays on the concrete
  =*oanda.Client= (or a =DataProvider= field, once that interface exists)
  until the =DataProvider= split happens — same deferral as Phase 1/2.

** Ledger merges into Account too

Decided 2026-07-22, prompted by a real-world sanity check: in standard
bookkeeping, a *ledger is something an account has* — the record of its
transactions — not a container an account lives inside. =account.Account=
already keeps most of its own ledger directly: =Trades []*Trade= (closed
transaction history), =Lots=/=AddLot=/=CloseLot= (position bookkeeping).
What the separate =account.Ledger= wrapper (=account/ledger.go=, the
struct Phase 1 renamed from =account.Broker=) adds on top is really just
the event queue (=evtQ=, =Events()=, =publishEvent=) and =SubmitOpen=/
=SubmitClose= as entry points wrapping =AddLot=/=CloseLot= with a
synchronous event fire — implementation plumbing, not a distinct
real-world concept.

Target: =account.Ledger= as a separate type goes away. Its narrowed,
correctly-scoped remaining job — a running record of transactions,
capital, and PnL, possibly order history — becomes something =Account=
*has* (e.g. a =Ledger= field/sub-struct on =Account=, not a struct that
wraps =Account=), not an orchestrator. =engine.Trader= (=engine/trader.go=,
currently =*account.Ledger= embedded) embeds =*account.Account= instead,
inheriting whatever of =Ledger='s current method set survives the merge.
=SubmitOpen=/=SubmitClose= as synchronous orchestration entry points do
*not* survive as-is either — see Phase 4, where order submission and fill
application become two separate, asynchronously-connected steps once the
event queue moves to =Broker=.

** Where the registry goes

=Service.accounts map[string]*Account=/=accountsMu=/=firstID= and the
=Account()=/=Accounts()=/=FirstAccount()=/=DefaultAccount()= constructor
methods move into =account/= alongside the merged type — e.g. an
=account.Registry= (or package-level functions closing over a registry
struct) that constructs/caches sessions by OANDA account ID. =service.Service=
keeps only a thin facade delegating to it, the same shape as
=service/review.go= delegating to =reviewsvc= today.

** Naming: =service/account= vs. =./account=

This phase does *not* introduce a new =service/account= package — the
session registry and merged =Account= type live directly in the existing
top-level =account/= package alongside =Ledger=/=Order=/=Trade=. No
same-package-name-different-import-path situation to manage (unlike
=review= vs. =service/review='s =reviewsvc= alias) — there is only one
=account= package once this lands.

* Phase 4: converge backtest and live onto one execution flow

Decided 2026-07-22, not yet implemented. Broader than Phase 3's type merge:
the goal is that backtest and live trading are *the same flow*, differing
only in which =Broker= is plugged in and where source-of-truth authority
sits — not two systems that happen to share a struct name. Explicit
principle from this discussion: *correctness over convenience* — this is a
real, larger refactor, accepted as such, to be taken one chunk at a time
rather than deferred for being big.

** The problem: backtest and live don't share a path today

- *Backtest*: =backtest/execute.go= computes the fill price itself (candle
  close, adjusted for spread/slippage via =account.FillAdjust(isBuy,
  candle.AvgSpread, slippage)=) and calls =Ledger.SubmitOpen=/=SubmitClose=
  directly. =SubmitOpen= trusts the caller's price outright and writes a
  =Lot= straight into =Account.Lots=. No =Broker= is ever consulted — the
  ledger write *is* the execution.
- *Live*: =service/orders.go= calls =a.broker().SubmitMarketOrder(...)=
  directly. OANDA executes and returns a real fill (which can differ from
  what was quoted). =account.Ledger=/=Account.Lots= is never touched — live
  has no local ledger today, relying entirely on querying OANDA
  (=AccountSnapshot=, =GetOpenTrades=) for state.

These are two independent code paths that both happen to end up at
something named =Account=. Merging the type (Phase 3) doesn't converge the
*flow* — =orders.go= would still call =broker()= directly on one side while
=execute.go= calls =Ledger.SubmitOpen= directly on the other.

** Target flow: orders are requests, fills are events

Reshaped 2026-07-22 from the original "=Ledger.SubmitOpen= calls =Broker=
synchronously" sketch. The trigger: a stop-loss is a *resting* order
sitting on the broker's book — it doesn't fire when you submit it, it
fires later, whenever price actually reaches it, and only the =Broker=
knows when that happens. A market order is the degenerate case of the same
idea (the "delay" is ~0), not a fundamentally different mechanism. So
submitting an order and applying its fill to an account's ledger are two
separate steps, connected asynchronously — not one synchronous call.

- *Submission*: something (chunk 3's live-trading migration, or backtest's
  per-bar loop) calls =Broker.SubmitMarketOrder(ctx, accountID, ...)= to
  open, or sets a resting stop/take via =Broker.UpdateTradeStop=. =Broker=
  addresses accounts by ID (already true of every method in the phase-1
  interface) — it does not need an =Account= handle to do this, and
  =Account= does not hold a =Broker= field (see Phase 3's correction).
- *Fill application*: =Broker.StreamTransactions= (currently one of chunk
  1's deliberately-minimal =Sim= methods — see chunk 1 below) becomes the
  *real*, unified fill feed for both implementations: =oanda.Client=
  already streams real fills; =Sim= needs to actually implement it,
  emitting when =SubmitMarketOrder= fills immediately *or* when a resting
  stop/take it's tracking gets hit on a later =UpdatePrice= call. Something
  subscribes to this stream, translates =oanda.TxEvent= → =account.Event=,
  and applies it to the account's ledger (=AddLot=/=CloseLot=, now
  triggered by the event rather than called directly by =SubmitOpen=).
- *Stop/take triggering moves into =Sim=.* =backtest/exits.go='s
  =autoCloseExits= currently checks candle OHLC against
  =lot.Stop=/=lot.Take= and calls =SubmitClose= directly, outside any
  broker — doing the broker's job in the wrong place. That check belongs
  inside =Sim=, run against each =UpdatePrice=, mirroring how a real
  resting stop sits on OANDA's book and triggers on its own.

Backtest and live differ only in which =Broker= is plugged in and how it's
fed prices (candles via =TickFromCandle=, vs. real ticks) — not in how
fills get applied to an account, which is now the same event-consumption
step either way.

Known, explicitly parked difference: backtest replays candles, live
trading (real or paper) reacts to ticks. Not solved here — acknowledged as
a separate problem =brokers/sim= will need to handle, not a blocker to
this shape.

** Source-of-truth asymmetry — narrower than first thought

The flow converges, and this turns out to converge *authority* further
than the original sketch assumed too: if both =Sim= and real =oanda.Client=
apply fills to an account's ledger through the same
=StreamTransactions=-consuming step, there's no separate "write locally,
then reconcile" phase for trade fills at all — the event stream *is* the
single write path for both. What's left asymmetric is narrower:

- *Sim*: its own =StreamTransactions= emissions are authoritative by
  construction — nothing external to disagree with.
- *OANDA*: =StreamTransactions= (or an equivalent poll,
  =AccountSnapshot='s existing =GetAccountChanges= mechanism) is OANDA's
  own authoritative feed already — consuming it *is* the reconciliation,
  not a step bolted on after an eager local write. =AccountSnapshot='s
  remaining distinct job likely shrinks to continuously-updating figures
  that aren't tied to a specific fill event (unrealized PnL from price
  moves, margin) rather than reconciling the trade ledger itself — open
  detail for chunk 4, not resolved here.

** Risk management moves onto Account

=account.Account= already has an =RiskFraction= field, but only backtest
uses it (set once at construction, e.g.
=acct.RiskFraction = run.Request.RiskPct= in =backtest/executor.go=, never
touched again). Live's sizing math — stop distance, =quoteToUSDRate=, unit
sizing, =MaxUnits=/=MaxPositionUSD= caps — lives entirely inside
=service/orders.go='s =PlaceMarketOrder=, blind to the account beyond a
fetched equity number passed in per-call.

Target: this sizing logic becomes a method *on* =Account=, reading its own
=Equity=/=RiskFraction=/open =Lots= directly. Backtest and live then size
positions through the identical function given an identically-shaped
account, rather than two independent implementations that happen to agree
by coincidence. This is also the concrete reason =Account= needs to *own or
reference* risk management, not have it float in =service/=: risk
calculations are inherently dependent on account size and open positions,
which only =Account= actually has.

** Broker events converge too — now via =StreamTransactions=, not =Ledger.evtQ=

Original framing: "=Ledger.Events()= already exists, =live_runner.go=
should consume it too." Revised: the event queue itself moves — it's
=Broker='s job to know when a resting order triggers (see "Target flow"
above), so the queue belongs on =Broker= (via =StreamTransactions=), not
on =Ledger=/=Account=. =engine.Trader.StartBrokerEventHandler=
(=engine/trader.go=) and =service/live_runner.go= both end up consuming
the same kind of feed — a =Broker='s =StreamTransactions=, translated to
=account.Event= — rather than =live_runner.go= gaining a consumer for
=Ledger.Events()= as originally sketched. Same convergence goal, different
(now more accurate) mechanism.

** Sequencing (proposed, not yet agreed chunk-by-chunk)

Roughly dependency-ordered — later chunks build on earlier ones being
correct:

1. Make =brokers/sim= a real =Broker= (spread/slippage-aware fills from
   candle data). Foundational — nothing else can converge onto it until it
   exists. *Done, commit =472b398=* — six real methods
   (=SubmitMarketOrder=, =CloseTrade=, =UpdateTradeStop=, =GetOpenTrades=,
   =GetAccountSummary=, =GetAccountDetails=), four deliberately minimal
   ones split into =broker_stub.go=, plus =TickFromCandle= as the
   not-yet-wired foundation chunk 2 needs. Confirmed via
   =var _ Broker = (*sim.Sim)(nil)= in =brokers/broker.go=.
2. *Split into two, first half done.* =Sim.StreamTransactions= is real —
   emitting on immediate market-order fills *and* on resting stop/take
   triggers it now monitors internally via =UpdatePrice= (moved out of
   =backtest/exits.go='s =autoCloseExits=, which still runs unchanged for
   now — this half was scoped self-contained to =brokers/sim=, touching
   nothing else). *Done, commit =ebc5625=* — six methods gained event
   emission, =checkStopsAndTakes= added, full long/short × stop/take test
   coverage.
3. *Done, commit =59e4869=.* Merge =account.Ledger= into =account.Account=
   (see Phase 3's "Ledger merges into Account too") — purely structural,
   zero behavior change. =SubmitOpen=/=SubmitClose= and the event-queue
   machinery moved to =account/events.go= as =*Account= methods;
   =account.Ledger=/=NewLedger= deleted; =engine.Trader= embeds
   =*account.Account= directly and gained a nil-safe =Broker
   brokers.Broker= field for chunk 4 to populate;
   =backtest/executor.go='s =BrokerFactory= removed entirely (no separate
   =Ledger= left to construct). Full test suite green, confirming nothing
   about *what happens* on open/close changed — only which type the
   methods live on.
4. *Not yet re-planned in detail — the highest-risk step.* Build the
   =StreamTransactions=-consuming step that applies fills to an account's
   ledger, replacing =SubmitOpen=/=SubmitClose='s synchronous
   =AddLot=/=CloseLot= calls, and wire =engine.Trader=/backtest onto real
   =Broker= calls via the new =Trader.Broker= field. This absorbs what was
   chunk 2's second half *and* most of what was chunk 6 (event
   convergence) — they turned out to be the same mechanism, not
   sequential steps. A regression here silently corrupts backtest results.

   *Baseline caveat (2026-07-22):* "bit-for-bit equivalence with current
   backtest behavior" is the right bar, but "current" must mean a *fresh*
   baseline captured on today's =main= immediately before this chunk
   starts — not a comparison against old saved results. The candle/
   timestamp cleanup (removing =CandleTime=) landed after the original
   backtest runs, so historical numbers may have already drifted for
   reasons unrelated to this refactor; comparing against them would make
   that pre-existing drift indistinguishable from a real regression. Run
   the existing backtest test suite plus a few of the real
   =testdata/configs/*.yml= backtests on =main= first, save that as the
   reference, then diff post-change output against *that*.
5. Migrate live (=orders.go=/=live_runner.go=) onto the same submit +
   consume-=StreamTransactions= flow, using =brokers/oanda=.
6. =AccountSnapshot='s role once fills flow through =StreamTransactions=
   uniformly — likely narrows to continuously-updating figures (unrealized
   PnL, margin) rather than trade-ledger reconciliation; see "Source-of-truth
   asymmetry" above. Exact scope still open.
7. Move risk/sizing math onto =Account=, shared by backtest and live.

Chunk boundaries and order still open for discussion — this is a first
pass, not a committed plan. Chunk 4 specifically needs a proper plan-mode
session (how =account.Event=/=oanda.TxEvent= translation should work,
exactly where the consuming step lives, and the fresh-baseline caveat
above) before implementation.
