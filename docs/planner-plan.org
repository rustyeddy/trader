#+title: Planner Layer — Incremental Plan
#+author: Claude
#+date: 2026-06-24
#+startup: overview

* Goal

Move the trading business logic out of the strategy and the run loop into a
dedicated =planner= layer, so the pipeline becomes:

#+begin_src text
strategy  →  Signal  →  Planner  →  Trader
  (pure alpha:        (sizing, risk,    (broker
   price action,       regime, spread,   submit,
   indicators)         stop placement)   events)
#+end_src

Strategies should emit pure *signals* (intent: side / strength / reason).
A *planner* owns all the order-construction business logic. The engine
(=Trader=) stays pure mechanism. One planner serves both =backtest= and =live=.

* Why

Today the planner responsibilities are smeared across two places:

- Strategies (e.g. =emacross.EmitOpen=) already decide side, compute the stop,
  build the =OpenRequest=, AND close opposing lots (reversal logic).
- The run loop layered on regime gating, spread gating, =exit.InitialStop=, and
  =account.SizePosition=.

=StrategyPlan.Opens= are =OpenRequest= values (side/price/stop/take/units) —
orders, not signals. That blur is what this refactor removes.

* Constraint (the safety rail)

Backtest report filenames are =<run-name>-<config-hash>= and regression
comparisons use *exact integer equality*. Every increment must be a
behavior-preserving extraction first: identical order sequencing → identical
results. Verify with =go test ./...= and =make sweep= (and the regression /
blackbox tests) BEFORE changing any interface.
