** Decision

Add =Timestamp types.Timestamp= directly to =market.Candle=. Delete
=candleTime=/=CandleTime= entirely. Every current use of =CandleTime=
becomes a use of =Candle= (which is now a strict superset — same fields
plus =Timestamp=). This is primarily a mechanical rename, not a
restructuring — see the API-stability constraint below.

** Why (full rationale — this was a considered tradeoff, not a default)

*** The problem this solves

The =candle-store-timestamp-fix= work (issue #179) exists because a
function had a true OANDA timestamp in hand and had nowhere structurally
appropriate to keep it attached to its candle all the way to disk — the
plain =market.Candle= type has no timestamp field, so timestamps were
being separately reconstructed (and reconstructed WRONG) at read time
instead of carried as data. The immediate #179 bug is fixed, but the
underlying gap — two candle types, one timeless and one timestamped,
with call sites choosing between them by convention rather than by
compiler-enforced correctness — remains, and is exactly the kind of gap
that produces this class of bug again later.

*** Alternative considered and rejected: keep Candle timeless, widen CandleTime's use at the store boundary only

This was seriously considered. It has a real advantage: it lets =Candle=
formally guarantee "no timestamp exists here," which is a stronger
static guarantee than "the timestamp field happens to be zero." It was
rejected on an explicit priority call: this is a long-lived,
single-maintainer codebase where minimizing the number of concepts a
future reader (including a future implementation agent) has to hold in
their head matters more than type-level purity. Two candle-shaped types
whose split exists to enforce one invariant is a second concept to
maintain, a wrapper type to keep alive, and conversion points at every
boundary between "the pure math side" and "the timestamped side" — and
in practice call sites already choose between them by convention (as
today's =CandleTime= usage shows), so the enforcement the split was
supposed to buy was already leaking before this decision.

The chosen alternative accepts a weaker guarantee (a zero =Timestamp=
on a synthetic/test candle just means "this candle is genuinely
timeless," by convention, not by the type system) in exchange for one
type, no conversions, no wrapper. Indicator math and other genuinely
time-agnostic code simply ignores the field — nothing forces it to, but
nothing needs it to.

*** RAM cost was evaluated and accepted, not ignored

+8 bytes/candle. Rough worst case (full ~21-year M1 history × ~24 pairs
resident in the process-lifetime cache simultaneously — a scenario no
current workflow triggers): on the order of 1.2–1.5 GB. Real, bounded,
and not a driving concern relative to the maintainability goal.

*** Governing principle (carries over from the #179 design addendum): timestamp is truth, index is position

The dense-array canonical store keeps its slot-per-timeframe-increment
model exactly as-is. The array position answers "where does this candle
live and in what order"; the stored =Candle.Timestamp= answers "when
did it actually open." These are deliberately separate questions after
this change — under the OLD (buggy) design the array position WAS the
timestamp (=Start + idx*step=), which is precisely how #179 happened.
This merge is what makes "timestamp is truth" hold everywhere, not just
in the store package.

** Hard constraint: API stability

The public shape of how code interacts with candles must not change,
beyond a mechanical symbol swap. Concretely:

- Everywhere that currently takes/returns =[]market.Candle= keeps doing
  so — the element type gains one field, nothing about call shape
  changes.
- Everywhere that currently takes/returns =CandleTime= (iterators,
  =CandleIterator.Next() (CandleTime, bool)=, strategy =Tick=/=Update=
  signatures, etc.) switches to =Candle=. Since =Candle= is now a strict
  superset of the old =CandleTime=, this is a symbol rename at call
  sites, not a logic change: =candleTime.Timestamp= access becomes
  =candle.Timestamp= access, =candleTime.Candle.Open= access becomes
  =candle.Open= access directly (one less level of embedding to walk
  through, if anything simpler).
- =CandleSet=, slot math, valid-bit handling, monthly-file sizing:
  unchanged by this part of the work (already correct post-#179).

Acceptance bar: outside =market/= and the small number of packages that
literally define candle iteration/storage (=datamanager/=), diffs
across the ~39 affected files should be overwhelmingly =CandleTime= →
=Candle= symbol substitutions and import cleanup. If implementing this
requires restructuring logic in =review/=, =strategy/=, or
=strategies/*= beyond a rename, stop and reassess — that means the
merge is being done wrong, or a bug is being uncovered along the way
(report it, don't quietly fix-and-continue inside this PR).

* Part 2 — Rejection-candle detection as a new pattern-recognition package

** Sequencing

Depends on Part 1 being landed and green. Interfaces below are written
directly against the merged =market.Candle= (with =.Timestamp=), not
against =CandleTime=.

** What this is, and why it is not an "indicator"

=indicator/= (existing package: =ADX=, =ATR=, =EMA=,
=ChoppinessIndex=, =BollingerBands=) has a consistent shape: each type
is an incremental, Wilder-style *accumulator* that consumes candles one
at a time via =Update(c market.Candle)=, grows smoothed state across the
full history it has seen, and exposes a continuous numeric read-out
(=Float64()=, =Value()=) answering "how strong / where / what regime,
right now, given everything so far." That is a specific, and
specifically continuous-valued, category.

Rejection-candle detection answers a different KIND of question: "does
THIS bar's shape (long wick, small body, close position within the
bar's range) match a known price-action pattern," evaluated from a
small fixed window (one bar, or a short lookback), not accumulated
history. This is categorical/pattern-matching, not continuous/regime.
It is a standard, separately-named category in technical analysis
generally (e.g. TA-Lib splits "Pattern Recognition Functions" out from
its momentum/trend/volatility indicator categories) — this is not an
arbitrary split invented for this codebase, it matches how the domain
is normally organized.

Consequence: this does NOT belong inside =indicator/=. It gets its own
package.

** Package: candlepattern/ (sibling to indicator/)

No existing package name collision (checked). =candlepattern/=,
matching the specificity of =indicator/='s own naming (plain =pattern=
was considered and rejected as too generic/likely to collide with
future non-candle pattern work, e.g. chart patterns, order-flow
patterns).

package candlepattern

// RejectionDetector recognizes rejection-candle price action from a
// short window of recent candles. Unlike indicator/'s accumulators,
// implementations hold only a small fixed lookback, not full history.
type RejectionDetector interface {
    Name() string

}

}

** First concrete implementation: WickRejection

Four tunable parameters (design already established 2026-07-15, carried
forward unchanged):

| param          | meaning                                                      |
|----------------+--------------------------------------------------------------|
| min-wick-ratio | wick length / total bar range, minimum to count as a wick    |
| max-close-pos  | close position within the bar's range, maximum (close must   |
|                | sit near the opposite end from the rejected wick)            |
| min-wick-atr   | minimum wick length, expressed in ATR multiples — filters    |
|                | out noise-scale wicks on quiet bars                          |
| lookback       | optional; >1 allows the pattern to be judged against a short |
|                | recent window rather than a single bar in isolation          |

#+begin_src go
package candlepattern

type WickRejection struct {
    minWickRatio float64
    maxClosePos  float64
    minWickATR   float64
    lookback     int

    atr    *indicator.ATR   // dependency: min-wick-atr needs a volatility
                            // reference; same relationship D1ADXFilter has
                            // to indicator.ADX
    window []market.Candle  // ring buffer, len <= lookback
    matched bool
    side    types.Side
}

func NewWickRejection(minWickRatio, maxClosePos, minWickATR float64,
    lookback int, atrPeriod int, scale types.Scale6) (*WickRejection, error)

func (w *WickRejection) Name() string { return "wick-rejection" }
func (w *WickRejection) Ready() bool  { return w.atr.Ready() && len(w.window) >= w.lookback }
func (w *WickRejection) Update(window []market.Candle) {
    // 1. feed the latest candle to w.atr (needs its own Update call —
    //    ATR is a full-history accumulator, unlike the pattern window)
    // 2. maintain the ring buffer at len == w.lookback
    // 3. evaluate min-wick-ratio / max-close-pos / min-wick-atr against
    //    the most recent candle(s) in the window
    // 4. set w.matched, w.side
}
func (w *WickRejection) Matched() bool    { return w.matched }
func (w *WickRejection) Side() types.Side { return w.side }
#+end_src

Note the ATR dependency's =Update= is called on every bar regardless of
=lookback= (ATR needs continuous history to stay warm), while the
pattern-matching window itself only needs the last =lookback= candles —
these are two different internal cadences inside the same type, worth a
code comment explaining why so a future reader does not "simplify" the
ring buffer into feeding ATR too.
