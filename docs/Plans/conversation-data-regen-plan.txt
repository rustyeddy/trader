Assistant: Investigated further and revised the plan:

- CandleTime insight (root cause 1): readCSVUncached already parses the
  true per-row timestamp from CSV column 0, but only uses it to compute
  idx, then discards it (store.go:561-566). WriteCSV synthesizes
  openUnix := cs.Start + i*step instead of writing a real timestamp
  (store.go:692); every downstream reader (candleSetIterator.Timestamp(),
  rangedCandleIterator.Next() in data_iterator.go:98) reconstructs time
  the same way. Since candle spacing is provably uniform, the fix doesn't
  need a hand-rolled NY/DST formula at all: DeriveCanonicalFromRaw should
  carry market.CandleTime (Candle + the real r.Time) through, and set
  cs.Start from the actual first observed raw timestamp (back-computed to
  slot 0), not from an assumed or computed alignment rule. This removes
  the DST-math helper from root cause 1 entirely. The cross-month raw-read
  boundary issue found earlier still applies regardless of this change and
  needs to be resolved in the same phase.

- H1/D1 derivation confirmed real (not hypothetical): Plan() in
  data_manager.go (lines 212-222 for H1<-M1, 232-242 for D1<-H1) creates
  build tasks whenever the lower timeframe is present in inventory, wired
  through candleMaker() (lines 288-289) into buildH1/buildD1, both calling
  cs.Aggregate() -> candle_agg.go's buggy bucketing. Checked the math:
  H1-from-M1 is provably unaffected (M1's Start is already UTC-hour-
  aligned; flooring to the hour is a no-op). D1-from-H1 is genuinely
  broken (flooring UTC-midnight Start to a day boundary collapses to UTC
  midnight instead of true NY 17:00, and there's no raw ground truth to
