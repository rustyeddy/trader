This document is a design plan; most of it describes shipped behavior as of
2026-07-11, with two sections still TODO (mean reversion classification;
config-driven thresholds — see below). It describes the data model, indicator
selection, triage logic, package placement, API shape, and CLI and UI
surfaces for a multi-timeframe watchlist scanner.

** Purpose and daily workflow

The scanner answers two questions every morning:

1. Which pairs are in a favorable regime? (promote to or retain on hotlist)
2. Of those, which have pulled back or compressed into a setup zone? (mark
   tradeable)

The output is a ranked table of all watchlist pairs with enough computed
context to decide in seconds whether a pair deserves a chart look. It does
not make trade decisions; it eliminates pairs that clearly do not merit
attention and surfaces the ones that do, in priority order.

** Timeframe stack: W1 → D1 → H4 (+ H1 for Tradeable entry timing)

The existing methodology in =FX-pair-analysis.org= specifies
three timeframes: D1 and H4 as primary classification inputs, H1 for
confirmation context only. The scanner implements that stack.

| Timeframe | Role                                                         |
|-----------+--------------------------------------------------------------|
| W1        | Big-picture trend direction; filters counter-trend D1 setups |
| D1        | Primary regime signal; ADX, CI, EMA alignment, volatility    |
| H4        | Setup quality; pullback structure, squeeze, entry proximity  |
| H1        | Entry-timing refinement for Tradeable pairs only — never a classification input, never computed for Watch/Hot |

H4 earns its place because D1 alone cannot distinguish between two hotlist
pairs where one has a clear pullback structure and the other is mid-range
with no setup. H4 CI and H4 EMA proximity answer that question without
requiring a chart.

*H1 (shipped, GitHub issue #166).* H1 is computed only for pairs D1/H4/W1
already classified "tradeable" — =Classify()= runs and returns first;
=review.EnrichWithH1= is called afterward, in the same scan pass, and is a
structural no-op for any other bucket. It cannot reinforce, challenge, or
change the bucket a pair already received. The indicator set is
deliberately lean (EMA20/EMA50/ATR only, no ADX/CI) since H1's only job is
"where, within an already-confirmed setup, is price relative to the H1
trend" — =SetupSnapshot.H1Aligned=/=H1EntryDist=. Missing/insufficient H1
history is best-effort: the pair still returns as tradeable, with a "H1
unavailable" note, never an error or a skip. Because this only fires for
the (typically small) tradeable subset of the watchlist, not the full ~20
pairs, the fetch-count growth this section originally worried about is
bounded by the tradeable bucket's size, not the watchlist's.

** Triage buckets

#+begin_src text
Watch     -- on the watchlist; no favorable regime signal yet
Hot       -- regime conditions are favorable; monitor closely
Tradeable -- on the hotlist and price is in a setup zone
#+end_src

Pairs move between buckets based on indicator thresholds evaluated at scan
time. There is no persistent state in v1 — each scan re-evaluates the full
watchlist from scratch.

** Indicator set

*** W1 indicators (higher-timeframe filter)

*Weekly EMA(20)* — already wired via =strategy.WeeklyEMAFilter=. Price above
the W1 EMA = bullish higher-timeframe bias; below = bearish. Filters out
counter-trend D1 setups.

*Weekly bias alignment (tri-state)* — the W1-vs-D1 relationship is not a
simple match/no-match. A flat W1 EMA (price hugging the weekly average,
biasFromEMA returning "neutral") is a genuinely different situation from a
directional W1 that confirms D1, and both are different from a directional
W1 that opposes D1. Collapsing all three into a single aligned/not-aligned
bool erases that distinction — a pair sitting in Hot because its W1 is flat
(not fighting, but not confirming either) looked identical in early
iterations of this scanner to a pair whose W1 actively confirms the D1
trend. =review/classify.go= implements this as a three-valued =Alignment=
type:

#+begin_src go
type Alignment string

const (
    Aligned  Alignment = "aligned"  // both directional and matching
    Neutral  Alignment = "neutral"  // higher timeframe is flat
    Conflict Alignment = "conflict" // both directional and opposed
)

func WeeklyAlignment(w1Bias, d1Bias string) Alignment
#+end_src

This is the single source of truth: the Hot gate's "weekly not fighting
D1" check, the demotion note, and the CLI's W1 column all derive from this
one function, so they can never silently disagree with each other.

*Weekly ATR(14)* — the expected range for the full week. If the pair has
already traveled 80%+ of its average weekly ATR, new entries on the remaining
week have poor reward-to-risk.

*** D1 indicators (regime classification)

*ADX(14) with ±DI* — the canonical trend-strength indicator, already
implemented as =indicator.ADX= with Wilder smoothing. ADX measures how
strongly price is trending; ±DI gives direction simultaneously. All three
values (ADX, +DI, −DI) are surfaced because they drive both regime
classification and directional bias in one pass.

*Choppiness Index(14)* — =indicator.ChoppinessIndex=, already implemented.
A direct measure of consolidation: high CI means the pair is oscillating
within a range, low CI means it is extending directionally. ADX and CI are
complementary, not redundant: ADX can be elevated while CI is also elevated
(a whippy trending pair), and the combination catches that case. Threshold:
< 55 trending, > 65 ranging.

*EMA(20) and EMA(50) on D1* — =indicator.EMA=, already implemented. Used
for trend alignment (price above both = bullish), EMA separation as a trend
quality signal, and price-to-EMA distance as an entry proximity measure.

*EMA separation ((EMA20 − EMA50) / ATR14)* — not a standalone indicator
but a one-line derived value. When the fast EMA is well separated from the
slow EMA, the trend is established and maturing. When they converge or cross,
the trend is weakening or reversing. Expressed in ATR multiples so it is
comparable across pairs with different pip scales (GBPJPY vs EURUSD). The
sign encodes direction; the magnitude encodes trend maturity.

*ATR(14) on D1* — =indicator.ATR=, already implemented. Serves as the
denominator for all ATR-normalized distance computations and as a standalone
volatility reference in pips for position sizing.

*Bollinger Bands(20, 2.0) on D1* — =indicator.BollingerBands=, already
implemented. Adds two signals the EMA set does not capture cleanly:

- *%B* — where price sits within the bands (0 = lower band, 1 = upper band).
  Values at the extremes indicate either momentum continuation or pending
  mean reversion.
- *Band width / ATR* — when narrow relative to ATR, the pair is in a
  volatility squeeze. Squeezes often precede explosive directional moves and
  are a strong setup filter.

*Body/range ratio (20-bar lookback)* — =datamanager.TrendAnalyzer=, already
implemented. The percentage of recent D1 bars where |Close − Open| / (High
− Low) > 0.6. A pair where 60% or more of recent daily bars are directional
is behaving more cleanly than one where bars are indecisive, regardless of
what ADX says.

*Price-to-EMA20 distance ((Close − EMA20) / ATR14)* — the primary setup
entry proximity signal. When a hotlist pair pulls back to within 0.5–1.5 ATR
of the D1 EMA20, it is in a value zone for trend-following entries.

*** H4 indicators (setup confirmation)

*H4 ADX(14)* — confirms the D1 trend is still alive at the shorter
timeframe. A pair with D1 ADX = 28 but H4 ADX = 14 may have lost momentum
at the entry timeframe.

*H4 Choppiness Index(14)* — catches consolidation that is not visible on D1.
A pair can look clean on the daily but be grinding sideways on H4, which
reduces setup quality significantly.

*H4 EMA(20) and price-to-EMA20 distance* — the H4 EMA distance is the
sharpest setup-proximity signal because H4 is the entry timeframe. Price
within 0.5–1.0 ATR of the H4 EMA20 with D1 trend still intact is the
canonical swing entry location.

*H4 EMA(50) and H4 EMA separation ((EMA20 − EMA50) / H4 ATR14)* — mirrors
the D1 EMA-separation signal, added specifically to distinguish a live H4
trend from a flattened/merged EMA20-EMA50 pair sitting near each other
during a consolidation. Before this field existed, H4 CI < 60 and
H4-EMA-proximity were the only setup-quality checks, and neither reliably
catches a multi-week range: a mature consolidation can score under 60 on
CI and land price near a flat, converged EMA20 by construction, not
because a real pullback happened. See "Known false-positive pattern"
below.

*H4 ATR(14)* — used for normalizing H4 distances; also confirms whether H4
volatility is compressed (squeeze) or expanding. Squeeze threshold as
implemented: =H4 BB width (upper − lower) < 2.0 × H4 ATR(14)=. This is a
hardcoded constant in =review/pair.go:computeH4= — not yet exposed as
config. 2.0×ATR is deliberately loose (Bollinger width at 2σ typically
exceeds 2×ATR in normal-volatility regimes), so the flag catches genuine
compression rather than triggering on ordinary noise.

*** Indicators excluded from v1

| Indicator    | Reason excluded                                            |
|--------------+------------------------------------------------------------|
| RSI          | Redundant over %B + ADX ±DI; adds threshold-tuning burden  |
| MACD         | EMA separation gives same information with less lag        |
| Stochastic   | Mean-reversion tool; wrong fit for trend-following scanner |
| Volume(1)    | OANDA spot FX tick volume is not meaningful                |
| Pivot points | Static levels; handled by S&R detection pipeline (v2)      |

1: It maybe a decent proxy for volume?

** =ReviewResult= struct

Note: this doc originally specified =ScanResult=/=ScanPair=/package
=scanner=. The shipped implementation lives in package =review=
(=review/review.go=, =review/pair.go=, =review/classify.go=) as
=ReviewResult= / =ReviewPair()=. The struct below reflects the actual
shipped shape; naming in the Service/REST/CLI sections further down has
not been cross-checked against source and may still say Scan* — see Open

#+begin_src go
// ReviewResult is the per-pair output of one watchlist review run.
// All price-scaled values are converted to float64 for presentation.
// ATR-normalized values are dimensionless ratios.
type ReviewResult struct {
    Instrument string    `json:"instrument"`
    ScannedAt  time.Time `json:"scanned_at"`

    // Triage bucket.
    Bucket string `json:"bucket"` // "watch" | "hot" | "tradeable"

    // Directional bias from indicator combination.
    Bias string `json:"bias"` // "long" | "short" | "neutral"

    // Multi-timeframe snapshots.
    W1    W1Snapshot    `json:"w1"`
    D1    D1Snapshot    `json:"d1"`
    H4    H4Snapshot    `json:"h4"`
    Setup SetupSnapshot `json:"setup"`

    // Human-readable notes for tooltip (e.g. "ADX rising, H4 squeeze").
    Notes []string `json:"notes,omitempty"`
}

type D1Snapshot struct {
    ADX     float64 `json:"adx"`
    PlusDI  float64 `json:"plus_di"`
    MinusDI float64 `json:"minus_di"`
    CI      float64 `json:"ci"`
    ATRPips float64 `json:"atr_pips"`
    EMA20   float64 `json:"ema20"`
    EMA50   float64 `json:"ema50"`
    Close   float64 `json:"close"`

    // Derived.
    EMASepATR     float64 `json:"ema_sep_atr"`    // (EMA20-EMA50)/ATR14
    PriceEMA20ATR float64 `json:"price_ema20_atr"` // (Close-EMA20)/ATR14
    BBPctB        float64 `json:"bb_pct_b"`
    BBWidthATR    float64 `json:"bb_width_atr"`
    TrendPct      float64 `json:"trend_pct"` // % of last 20 bars trending
}

type H4Snapshot struct {
    ADX           float64 `json:"adx"`
    CI            float64 `json:"ci"`
    ATRPips       float64 `json:"atr_pips"`
    EMA20         float64 `json:"ema20"`
    EMA50         float64 `json:"ema50"`
    Close         float64 `json:"close"`
    PriceEMA20ATR float64 `json:"price_ema20_atr"` // (Close-EMA20)/H4 ATR14
    Squeeze       bool    `json:"squeeze"`          // H4 BB width < 2.0x H4 ATR

    // Derived. Mirrors D1.EMASepATR; distinguishes a live H4 trend from a

** Bucket classification logic

Evaluated in order; the first matching bucket wins.

#+begin_src text
Tradeable:
  bucket qualifies as Hot (passes Hot gate below)
  AND abs(H4.PriceEMA20ATR) in [0.5, 1.5]   -- near H4 EMA (entry zone)
  AND H4 bias matches D1 bias                -- short-term aligned with regime
  AND H4.CI < 60                             -- H4 not actively ranging
  AND H4.ADX >= 20                           -- H4 momentum still alive
  AND abs(H4.EMASepATR) >= 0.3               -- H4 EMA20/50 not merged/flat

Hot:
  D1.ADX >= 25
  AND D1.CI < 55
  AND WeeklyAlignment(W1 bias, D1 bias) != Conflict  -- Aligned or Neutral both pass;
                                                       -- only a directional W1 opposing
                                                       -- D1 blocks promotion

Watch:
  everything else
#+end_src

The two new Tradeable-only checks (=H4.ADX >= 20=, =abs(H4.EMASepATR) >=
0.3=) do *not* apply to the Hot gate — a pair can still be legitimately
Hot with a consolidating H4. This is intentional: Hot means "the D1/W1
regime is favorable," not "the H4 entry structure is clean." Only
Tradeable claims the entry structure itself is valid, so only Tradeable
enforces it. See "Known false-positive pattern" below for why these two
checks exist.

Demotion signals (surfaced as notes in v1, enforced as hard gates in v2 —
except the two Tradeable-only checks above, which are hard gates already):

#+begin_src text
ADX dropped below 20
CI crossed above 65
WeeklyAlignment(W1 bias, D1 bias) == Conflict   -- "W1 EMA flipped against D1 bias"
H4.ADX < 20                                     -- surfaced as a note even for Hot pairs
abs(H4.EMASepATR) < 0.3                         -- surfaced as a note even for Hot pairs
W1.WeekUsedPct >= 0.90                          -- weekly ATR budget consumed
#+end_src

** Known false-positive pattern: H4 consolidation vs pullback

*Diagnosis (2026-07-09).* Across multiple independent chart-review
sessions covering NZDUSD, AUDUSD, and USDCAD, every case where the scanner
called a pair Tradeable or Hot but chart review disagreed showed the same
signature: H4 ADX below 20 (observed as low as 13.46, consistently in the
18–21 range) combined with H4 EMA20/EMA50 converged or merged. In each
case the H4 structure was a multi-week sideways range, not a pullback
within a live trend — the daily regime was often genuinely strong (D1 ADX
in the high 30s to low 50s in several cases), but the entry-timeframe
structure had gone flat underneath it. The pre-existing checks (H4.CI <
60, H4 EMA proximity) did not catch this: a mature range can score under
60 on CI, and price sitting near a flattened EMA20 by construction is not
distinguishable from price sitting near an EMA20 mid-pullback without an
independent measure of whether the H4 EMAs are still separated.

A second, independent pattern showed up in the same review sessions:
pairs at or beyond their weekly ATR budget (USDCHF 91%, GBPAUD 138%,
GBPNZD 139%) were classified Tradeable/Hot without any flag, despite the
doc's own stated 80% caution threshold never being wired into
=demotionNotes=.

*Fix (this revision).* =H4.EMA50= and the derived =H4.EMASepATR= were
added (see H4 indicators, above) and Tradeable now hard-gates on
=H4.ADX >= 20 AND abs(H4.EMASepATR) >= 0.3=. =W1.WeekUsedPct >= 0.90= now
surfaces as a demotion note. Both threshold values are first-pass
estimates reasoned from the observed cases, not fit against a larger
sample — they should be revisited once more data accumulates.

*Calibration risk to watch.* As of this revision, zero pairs have been
observed as genuinely Tradeable across the full watchlist during the
entire period chart review has been running alongside the scanner. That
is plausibly correct — the methodology is deliberately selective and
low-frequency by design — but it is also exactly the pattern you'd see if
a gate were calibrated too tight. There is no way to distinguish those two
explanations from inside this codebase; it requires accumulating more
scan-and-chart-review sessions over time. If Tradeable remains at or near
zero for an extended period even as individual pairs show what look like
genuinely clean H4 pullbacks on the chart, that is the signal to revisit
=h4ADXFloor= and =h4MinEMASep= rather than assume the market simply hasn't
offered a setup yet.

** Reading the output against the chart

The scanner narrows the watchlist; it does not replace looking at the
chart. This section is a guide to what each signal typically *looks like*
on a chart, since the numbers alone can be misleading without that context.

*** Bucket ≠ setup quality

A pair passing every numeric gate can still be a watch-and-wait — this is
true for Hot even after the fix below, since Hot only checks the D1/W1
regime, not H4 entry structure. The canonical failure mode: Hot (or,
before this revision, sometimes Tradeable) by the numbers, but the H4
chart shows merged/flat EMA20 and EMA50 — a consolidation, not a
trend-pullback sequence. ADX and CI describe the *daily* regime; they
cannot see that H4 has gone sideways into a coil.

As of this revision, Tradeable hard-gates on =H4.ADX >= 20= and
=abs(H4.EMASepATR) >= 0.3= specifically to catch this (see "Known
false-positive pattern," above), so a merged-EMA H4 should no longer
reach Tradeable. Hot is deliberately left unfiltered on this — a Hot pair
with a consolidating H4 is exactly the state you want to be watching, not
one you'd want hidden. Use H4 DIST, H4 CI, and now H4 EMA sep together
with a glance at the H4 chart on Hot pairs; don't treat the absence of
H4-consolidation filtering on Hot as an oversight.

*** EMA separation sign vs. bias

=d1.ema_sep_atr= should normally agree in sign with =bias= (positive +
long, negative + short). If they disagree — EMA20 above EMA50 but bias
reads short, for example — the pair is very likely mid-transition: the
EMA structure hasn't caught up to a fresh ±DI cross yet. This is common and
not itself alarming, but worth a chart look before trusting either signal
in isolation, especially near a Hot/Watch promotion boundary. Recall bias
here is derived from close-vs-EMA20 (=biasFromEMA=), not from ±DI, so a
transitional pair can show conflicting EMA-sep sign and bias simultaneously
without anything being "wrong" — it just means the trend is young.

*** Reading Squeeze

A Squeeze flag (H4 BB width < 2.0×H4 ATR) means price has coiled tightly
on H4. On the chart this looks like flattening, converging bands with
small-range bars. It's a *coiled-spring* signal, not a directional one —
it tells you a move is likely, not which way. Pair it with W1 alignment
before assuming continuation:

- Squeeze + W1 Aligned → coiling in the direction of the higher-timeframe
  trend; the more constructive read, worth a close look for a breakout
  continuation.
- Squeeze + W1 Conflict → ambiguous; could resolve as continuation or as
  the start of a reversal. Treat as lower conviction until it resolves.

*** Reading W1 alignment (tri-state)

- *Aligned (✓)* — W1 and D1 point the same direction. The strongest
  regime confirmation; a Hot or Tradeable pair here has multi-timeframe
  agreement.
- *Neutral (~)* — W1 is flat (price hugging the weekly EMA20). This is
  not a red flag, but it is a lower-conviction context than Aligned: the
  weekly hasn't confirmed the daily trend yet. On the chart, look for
  whether W1 is flattening because it's about to turn (watch for it
  flipping to Conflict) or just consolidating before continuing.
- *Conflict (✗)* — W1 opposes D1. The pair can still be Hot or even
  Tradeable by the numeric gates (the Hot gate only excludes hard
  conflict, not neutral), but this is the case worth the most scrutiny on
  the chart — a D1 pullback fighting the weekly structure is a
  meaningfully different risk profile than one with a clean W1 tailwind,
  even inside the same bucket.

*** TrendPct and BBPctB

=d1.trend_pct= (body/range > 0.6 over the last 20 bars) is a cleanliness
check independent of ADX — a pair can have high ADX from a few large
directional bars mixed with choppy indecision, and TrendPct exposes that.
Low TrendPct alongside high ADX is worth a chart look even in a Tradeable
bucket.

=d1.bb_pct_b= isn't currently surfaced in CLI/UI output but is computed
and available on every =ReviewResult=. Values near 0 or 1 mean price is at
a band extreme — useful context when interpreting a Squeeze or an
extended EMA-distance reading, and directly relevant groundwork for the
mean reversion classification below.

** Data requirements

| Data                        | Candles needed | Warmup driver                             |
|-----------------------------+----------------+-------------------------------------------|
| W1 EMA(20), ATR(14)         | 30 W1          | EMA(20) warmup = 20                       |
| D1 ADX(14), ATR(14), CI(14) | 200 D1         | ADX *convergence* (not just Ready()); see below |
| D1 EMA(20), EMA(50), BB(20) | 200 D1         | EMA(50) warmup = 50; covers all           |
| H4 ADX(14), ATR(14), CI(14) | 200 H4         | Same convergence requirement as D1        |
| H4 EMA(20)                  | 200 H4         | EMA(20) warmup = 20; covered by the ADX-driven count |
| H1 EMA(20), EMA(50), ATR(14) (tradeable pairs only) | 200 H1 | Same convergence requirement, not just Ready() |
|-----------------------------+----------------+-------------------------------------------|

*D1/H4/H1 candle counts (GitHub issue #175).* These were 60 until an
empirical investigation found that `ADX.Ready()`'s `2*N=28`-period
threshold only guarantees the Wilder-smoothed value *has a number*, not
that it has *converged* to what a continuously-running series (like a
charting tool's, backed by years of history) would show at the same bar.
Synthetic testing found ~0.6-4.7 ADX points of drift at a 60-candle cold
start versus a long-run reference, dropping to <0.1 points by ~100-150
candles. 200 leaves headroom beyond that empirically-observed convergence
point. `reviewWeeklyLookbackDays` (340 calendar days) is sized to fetch
enough D1 history to cover this 200-candle window plus the 30-week W1
derivation.

For 20 watchlist pairs that is 60 OANDA candle requests (3 per pair, each
now a larger candle count than before, but the same request count). A
bounded goroutine pool of 8 workers handles the fan-out without rate-limit
risk. =CandlesOptions= already supports =Count= and =Granularity=. The H1
row adds a 4th request, but only for pairs that end up classified
tradeable in that scan — the worst case is bounded by the tradeable
bucket's size, not the full watchlist.

*** Candle fetch path

All candle fetching, present and future, goes through the existing
=DataManager= — never a scattered/ad-hoc fetch call inside a new package.
=review/= itself already follows this: =ReviewPair()= and its
=computeD1=/=computeH4=/=computeW1= helpers are pure functions that take
=[]market.Candle= as input; they do not fetch. Whatever wires
=Service.ScanWatchlist()= (or its =Review*=-renamed equivalent, see Open
Questions) to OANDA should call through =DataManager=, not open a
parallel fetch path. This applies equally to the mean reversion TODO
below — it reuses the same D1/H4 candles already fetched for trend
classification, so it introduces no new fetch surface at all.

** Service method

Shipped as =ReviewRequest=/=ReviewResponse=/=Service.ReviewWatchlist= in
=service/review.go= (confirmed against source 2026-07-11; this section
previously used speculative =Scan*=/=scanner= naming — see resolved Open
Question #6, below). =HotlistIDs= from the original sketch was dropped;
demotion notes are computed from the snapshot data alone.

#+begin_src go
// ReviewRequest parameterises a watchlist review run.
type ReviewRequest struct {
    // Instruments to review. Defaults to market.AllInstruments() (every
    // pair in the instrument registry) when empty.
    Instruments []string
}

// ReviewResponse is the full review output.
type ReviewResponse struct {
    ScannedAt time.Time             `json:"scanned_at"`
    Results   []review.ReviewResult `json:"results"`
}

// ReviewWatchlist runs the scanner over all instruments in req,
// fetches D1, H4, and W1 candles (via DataManager, OANDA-backed),
// computes all indicators, and returns a classified ReviewResponse.
func (s *Service) ReviewWatchlist(ctx context.Context, req ReviewRequest) (*ReviewResponse, error)
#+end_src

The service method is a thin fan-out over =reviewWorkers= (8) goroutines
per pair, each calling =review.ReviewPair()=, a pure function that takes
candle slices and returns a =ReviewResult=. The service layer owns
concurrency and fetching; the =review= package owns indicator composition
and classification.

** TODO Mean reversion classification (strategy map)

Sequencing note: per the trading plan, this comes *after* trend-following
is comfortable and after the equities expansion — added here as a design
placeholder so the shape is ready when that time comes, not as a v1
commitment.

*** Why this doesn't fit into the existing =Bucket= field

Trend-following and mean reversion want opposite regime conditions:

| Signal      | Trend-following (current)      | Mean reversion                    |
|-------------+---------------------------------+------------------------------------|
| D1 ADX      | >= 25 (trending)                | typically < 20 (no trend)          |
| D1 CI       | < 55 (extending)                | > 65 — currently the *exclusion*   |
| Entry       | pullback toward D1 EMA20        | extension away from mean, at a     |
|             |                                  | band extreme                       |
| BB %B       | not gated on                    | near 0 or 1 (band extreme)         |
| Bias        | trend direction                 | fade direction (opposite of        |
|             |                                  | extension)                         |
| Squeeze     | secondary continuation signal   | mostly irrelevant, possibly a      |
|             |                                  | caution flag (squeezes usually     |
|             |                                  | resolve into trends, not reverts)  |

A pair can legitimately be Watch for trend and Tradeable for MR at the
same time — a ranging AUDCAD at D1 ADX 14 is exactly the case
trend-following ignores that MR would want. This means a single =Bucket
string= can't represent both classifications; the struct needs to carry
results per strategy.

*** Struct shape: strategy map

#+begin_src go
// ReviewResult gains a Strategies map alongside (or replacing) the single
// Bucket/Bias fields, keyed by strategy name ("trend", "mr", ...).
type ReviewResult struct {
    Instrument string    `json:"instrument"`
    ScannedAt  time.Time `json:"scanned_at"`

    Strategies map[string]StrategyResult `json:"strategies"`

    W1    W1Snapshot    `json:"w1"`
    D1    D1Snapshot    `json:"d1"`
    H4    H4Snapshot    `json:"h4"`
    Setup SetupSnapshot `json:"setup"`
}

type StrategyResult struct {
    Bucket string   `json:"bucket"` // "watch" | "hot" | "tradeable"
    Bias   string   `json:"bias"`   // "long" | "short" | "neutral"
    Notes  []string `json:"notes,omitempty"`
}
#+end_src

This keeps =D1Snapshot=/=H4Snapshot=/=W1Snapshot= as shared raw
indicator data (computed once per pair regardless of strategy count) and
makes bucket/bias/notes strategy-scoped. Adding a third sub-strategy later
is a new map key, not a struct migration.

#+begin_src text
MR Hot:
  D1.ADX < 20                      -- no established trend
  AND D1.CI > 65                   -- actively ranging

MR Tradeable:
  qualifies as MR Hot
  AND abs(D1.BBPctB - 0.5) > 0.4   -- price near a band extreme (BBPctB
                                     -- near 0 or 1)
  AND NOT setup.squeeze            -- exclude coiling pairs; a squeeze here
                                     -- more often precedes a trend than a
                                     -- reversion

MR bias:
  BBPctB > 0.9  -> "short" (fade the upper-band extension)
  BBPctB < 0.1  -> "long"  (fade the lower-band extension)
  else          -> "neutral"

addition.

** Historical: backtestable review via =--asof= (implemented 2026-07-11)

*Motivation, as originally written.* Directly answers the calibration-risk
question raised in "Known false-positive pattern," above: as of that
revision, zero pairs had been observed as genuinely Tradeable, and there was
no way from inside the codebase to tell whether that reflected correct
selectivity or an over-tight =h4ADXFloor=/=h4MinEMASep=. Replaying the
classification logic against months of history was judged faster than
waiting for more real scan-and-chart-review sessions to accumulate.

*Status.* Fully shipped, in four ordered sub-projects, via PRs #152–#155.
The full design reasoning (why four sub-projects instead of one, the
=DataManager= in-memory cache, the bounded/compacted candle accessor, the
=service/review.go= refactor, and the no-lookahead semantics that were the
highest-risk item) is preserved in
[[file:archive/asof-review-sweep-spec.md][docs/archive/asof-review-sweep-spec.md]]
rather than duplicated here.

*Where it landed:*

| Spec section                          | Shipped as                                                        |
|----------------------------------------+---------------------------------------------------------------------|
| §1 DataManager in-memory candle cache | =datamanager/store.go= — process-lifetime =cache map[Key]*CandleSet= |
| §2 Bounded, compacted candle accessor | =datamanager/get_candles.go= — =DataManager.GetCandles(ctx, req, asof, count)= |
| §3 =service/review.go= refactor       | =service/review.go=, =service/review_sweep.go=                     |
| §4 =--asof= sweep                     | =service.ReviewWatchlistRange= + =ReviewRangeRequest=; CLI flags below |

*Semantics, as shipped.* =--asof= is a cutoff instant: only candles fully
closed as of that timestamp are eligible (closed-bars-only, no opt-out —
the no-lookahead constraint from spec §4.2). Dates parse as UTC midnight:
=--asof 2026-06-15= means "as of the start of June 15," i.e. using data
through June 14's close — simpler than the spec's speculative 17:00 NY
default, and unambiguous since it's always the same cutoff regardless of
session.

*CLI surface, as shipped* (=trader review=):

#+begin_src text
--asof YYYY-MM-DD              classify as of this past date instead of now;
                                cannot combine with --from/--to
--from / --to YYYY-MM-DD       historical sweep range (inclusive); must be
                                set together
--interval DURATION            step interval between sweep dates (default 24h)
--output csv|json|table         required to be csv or json for a multi-date
                                sweep; a single date may still use table
#+end_src

*What this is not.* Confirmed as originally scoped: this replays the
*classification layer only* — Bucket/Bias/Notes over time, not simulated
entries, stops, or P&L. See =/srv/trading/playbooks/playbook.org= for the
separate discretionary-entry layer that consumes Tradeable-bucket output,
and its TODO items 3–5 (composite score, config-driven thresholds, entry/
stop/size command) for what actually trading these signals would still
require.

*Follow-up — RESOLVED (2026-07-12, GitHub issue #165):* the classification
gates/thresholds (=H4ADXFloor=, =H4MinEMASep=, the D1 hot gate, H4 CI
ceiling, etc. — see "Bucket classification logic," above) are now
=review.Thresholds=, a struct threaded explicitly through =Classify()= and
=ReviewPair()= instead of hardcoded constants. Callers layer overrides
three ways, each optional: global config's =review:= section in
=trader.yml= (see =config.GlobalReviewConfig=), or one =trader review= flag
per threshold (=--h4-adx-floor=, =--hot-ci-ceiling=, etc.) for ad-hoc sweep
runs. An unconfigured field falls back to =review.DefaultThresholds()=, so
behavior is unchanged unless a value is explicitly overridden. This is what
actually answers the calibration-risk question this section originally
raised — the =--asof= sweep can now vary gates run-to-run without a
rebuild.

** Open questions

1. *Watchlist persistence.* In v1 the watchlist defaults to the 7 majors plus
   any pairs passed as a query parameter or CLI flag. A persistent watchlist
   and hotlist implies state that survives restarts. Options: extend the global
   config YAML with a =[watchlist]= and =[hotlist]= section, or store in a
   lightweight local file. The hotlist in particular needs to survive a restart
   since it represents a deliberate daily judgment.

2. *Scan scheduling.* The endpoint is on-demand (pull) in v1. A future step
   is a scheduled daily push: run at 17:00 NY close, cache the result in an
   in-memory field on =Service= (or a JSON file in the data directory), and
   serve the cached response until the next scheduled run. Pairs the REST and
   CLI endpoints would gain a =?cached=true= flag to serve the last scan
   without re-fetching candles.

3. *S&R levels as setup signal.* When the S&R detection pipeline
   (=analysis/=) is wired into the scanner, =SetupSnapshot= gains a
   =NearSRLevel bool= field and the Tradeable gate can require either EMA
   proximity *or* S&R proximity. The struct shape is designed to accommodate
   this without breaking the API.

4. *Integration with =FX-pair-analysis.org=* (corrected filename; the doc
   was previously referenced as =fx-pair-analysis-methodology.org=, which
   does not exist). That doc specifies a weighted quality scoring model
   (trend clarity 25%, structure clarity 25%, level clarity 15%, volatility
   fit 15%, macro alignment 10%, execution cleanliness 10%) and a three-tier
   watchlist prioritization scheme, still unreconciled with this doc's
   gate-based =Classify()=. Also relevant:
   =/srv/trading/playbooks/playbook.org= (the discretionary entry layer
   downstream of Tradeable) independently sketches a similar A+/B/Pass
   grading rubric and explicitly calls it "the discretionary ancestor" of a
   planned composite 0–100 score, intended to eventually consume labeled
   journal data to choose weights — the same unification gap as this item,
   approached from the entry-execution side rather than the scanner side.
   These three (this doc's hard gates, =FX-pair-analysis.org='s weighted
   score, the playbook's grading rubric) should converge rather than stay
   three independently-maintained models of the same judgment.

5. *Multi-account.* The review is market data, not account data, so it does
   not directly depend on multi-account support. Once multi-account is live,
   each account may maintain its own watchlist. =ReviewRequest= is
   parameterised by =Instruments= so this is solved at the call site without
   changing =review= internals.

6. *Naming drift between this doc and shipped code — RESOLVED (2026-07-11).*
   Confirmed against source: the service method, REST endpoint, and CLI
   command sections (above) were cross-checked and updated to
   =ReviewRequest=/=ReviewResponse=/=Service.ReviewWatchlist=/
   =GET /api/v1/review=/=trader review=. The original =Scan*=/=scanner=
   naming from the initial design was not carried into any shipped layer —
   =review= is the only package name used, end to end.

7. *Configurable gates/thresholds — RESOLVED (2026-07-12, GitHub issue
   #165).* =review/classify.go='s and =review/pair.go='s threshold
   literals now live in =review.Thresholds=, configurable via
   =trader.yml='s =review:= section and/or per-invocation =trader review=
   flags (flags win over config, config wins over
   =review.DefaultThresholds()=). =playbook.org='s TODO item 4 ("make X
   and Y config values from day one so the sweep can vary them without
   recompiling") is satisfied by the same change.
