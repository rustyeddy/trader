This document is a design plan, not yet-shipped behavior. It describes the
data model, indicator selection, triage logic, package placement, API shape,
and CLI and UI surfaces for a multi-timeframe watchlist scanner.

Related documents: [[file:fx-pair-analysis-methodology.org][fx-pair-analysis-methodology.org]],
[[file:multi-account.md][multi-account.md]], [[file:Services.md][Services.md]], [[file:ROADMAP.md][ROADMAP.md]].

** Purpose and daily workflow

The scanner answers two questions every morning:

1. Which pairs are in a favorable regime? (promote to or retain on hotlist)
2. Of those, which have pulled back or compressed into a setup zone? (mark
   tradeable)

The output is a ranked table of all watchlist pairs with enough computed
context to decide in seconds whether a pair deserves a chart look. It does
not make trade decisions; it eliminates pairs that clearly do not merit
attention and surfaces the ones that do, in priority order.

** Timeframe stack: W1 → D1 → H4

The existing methodology in =fx-pair-analysis-methodology.org= specifies
three timeframes: D1 and H4 as primary classification inputs, H1 for
confirmation context only. The scanner implements that stack.

| Timeframe | Role                                                         |
|-----------+--------------------------------------------------------------|
| W1        | Big-picture trend direction; filters counter-trend D1 setups |
| D1        | Primary regime signal; ADX, CI, EMA alignment, volatility    |
| H4        | Setup quality; pullback structure, squeeze, entry proximity  |

H4 earns its place because D1 alone cannot distinguish between two hotlist
pairs where one has a clear pullback structure and the other is mid-range
with no setup. H4 CI and H4 EMA proximity answer that question without
requiring a chart. H1 is available as a future refinement but is excluded
from v1 to keep the candle fetch surface minimal and the indicator warmup
requirements tractable.

** Triage buckets

#+begin_src text
Watch     -- on the watchlist; no favorable regime signal yet
Hot       -- regime conditions are favorable; monitor closely
Tradeable -- on the hotlist and price is in a setup zone
#+end_src

Pairs move between buckets based on indicator thresholds evaluated at scan
time. There is no persistent state in v1 — each scan re-evaluates the full
watchlist from scratch.

** Indicator set

*** W1 indicators (higher-timeframe filter)

*Weekly EMA(20)* — already wired via =strategy.WeeklyEMAFilter=. Price above
the W1 EMA = bullish higher-timeframe bias; below = bearish. Filters out
counter-trend D1 setups.

*Weekly bias alignment (tri-state)* — the W1-vs-D1 relationship is not a
simple match/no-match. A flat W1 EMA (price hugging the weekly average,
biasFromEMA returning "neutral") is a genuinely different situation from a
directional W1 that confirms D1, and both are different from a directional
W1 that opposes D1. Collapsing all three into a single aligned/not-aligned
bool erases that distinction — a pair sitting in Hot because its W1 is flat
(not fighting, but not confirming either) looked identical in early
iterations of this scanner to a pair whose W1 actively confirms the D1
trend. =review/classify.go= implements this as a three-valued =Alignment=
type:

#+begin_src go
type Alignment string

const (
    Aligned  Alignment = "aligned"  // both directional and matching
    Neutral  Alignment = "neutral"  // higher timeframe is flat
    Conflict Alignment = "conflict" // both directional and opposed
)

func WeeklyAlignment(w1Bias, d1Bias string) Alignment
#+end_src

This is the single source of truth: the Hot gate's "weekly not fighting
D1" check, the demotion note, and the CLI's W1 column all derive from this
one function, so they can never silently disagree with each other.

*Weekly ATR(14)* — the expected range for the full week. If the pair has
already traveled 80%+ of its average weekly ATR, new entries on the remaining
week have poor reward-to-risk.

*** D1 indicators (regime classification)

*ADX(14) with ±DI* — the canonical trend-strength indicator, already
implemented as =indicator.ADX= with Wilder smoothing. ADX measures how
strongly price is trending; ±DI gives direction simultaneously. All three
values (ADX, +DI, −DI) are surfaced because they drive both regime
classification and directional bias in one pass.

*Choppiness Index(14)* — =indicator.ChoppinessIndex=, already implemented.
A direct measure of consolidation: high CI means the pair is oscillating
within a range, low CI means it is extending directionally. ADX and CI are
complementary, not redundant: ADX can be elevated while CI is also elevated
(a whippy trending pair), and the combination catches that case. Threshold:
< 55 trending, > 65 ranging.

*EMA(20) and EMA(50) on D1* — =indicator.EMA=, already implemented. Used
for trend alignment (price above both = bullish), EMA separation as a trend
quality signal, and price-to-EMA distance as an entry proximity measure.

*EMA separation ((EMA20 − EMA50) / ATR14)* — not a standalone indicator
but a one-line derived value. When the fast EMA is well separated from the
slow EMA, the trend is established and maturing. When they converge or cross,
the trend is weakening or reversing. Expressed in ATR multiples so it is
comparable across pairs with different pip scales (GBPJPY vs EURUSD). The
sign encodes direction; the magnitude encodes trend maturity.

*ATR(14) on D1* — =indicator.ATR=, already implemented. Serves as the
denominator for all ATR-normalized distance computations and as a standalone
volatility reference in pips for position sizing.

*Bollinger Bands(20, 2.0) on D1* — =indicator.BollingerBands=, already
implemented. Adds two signals the EMA set does not capture cleanly:

- *%B* — where price sits within the bands (0 = lower band, 1 = upper band).
  Values at the extremes indicate either momentum continuation or pending
  mean reversion.
- *Band width / ATR* — when narrow relative to ATR, the pair is in a
  volatility squeeze. Squeezes often precede explosive directional moves and
  are a strong setup filter.

*Body/range ratio (20-bar lookback)* — =datamanager.TrendAnalyzer=, already
implemented. The percentage of recent D1 bars where |Close − Open| / (High
− Low) > 0.6. A pair where 60% or more of recent daily bars are directional
is behaving more cleanly than one where bars are indecisive, regardless of
what ADX says.

*Price-to-EMA20 distance ((Close − EMA20) / ATR14)* — the primary setup
entry proximity signal. When a hotlist pair pulls back to within 0.5–1.5 ATR
of the D1 EMA20, it is in a value zone for trend-following entries.

*** H4 indicators (setup confirmation)

*H4 ADX(14)* — confirms the D1 trend is still alive at the shorter
timeframe. A pair with D1 ADX = 28 but H4 ADX = 14 may have lost momentum
at the entry timeframe.

*H4 Choppiness Index(14)* — catches consolidation that is not visible on D1.
A pair can look clean on the daily but be grinding sideways on H4, which
reduces setup quality significantly.

*H4 EMA(20) and price-to-EMA20 distance* — the H4 EMA distance is the
sharpest setup-proximity signal because H4 is the entry timeframe. Price
within 0.5–1.0 ATR of the H4 EMA20 with D1 trend still intact is the
canonical swing entry location.

*H4 ATR(14)* — used for normalizing H4 distances; also confirms whether H4
volatility is compressed (squeeze) or expanding. Squeeze threshold as
implemented: =H4 BB width (upper − lower) < 2.0 × H4 ATR(14)=. This is a
hardcoded constant in =review/pair.go:computeH4= — not yet exposed as
config. 2.0×ATR is deliberately loose (Bollinger width at 2σ typically
exceeds 2×ATR in normal-volatility regimes), so the flag catches genuine
compression rather than triggering on ordinary noise.

*** Indicators excluded from v1

| Indicator    | Reason excluded                                            |
|--------------+------------------------------------------------------------|
| RSI          | Redundant over %B + ADX ±DI; adds threshold-tuning burden  |
| MACD         | EMA separation gives same information with less lag        |
| Stochastic   | Mean-reversion tool; wrong fit for trend-following scanner |
| Volume(1)    | OANDA spot FX tick volume is not meaningful                |
| Pivot points | Static levels; handled by S&R detection pipeline (v2)      |

1: It maybe a decent proxy for volume?

    EMA20       float64 `json:"ema20"`

    Squeeze     bool      `json:"squeeze"`       // H4 BB squeeze

** Bucket classification logic

Evaluated in order; the first matching bucket wins.

#+begin_src text
Tradeable:
  bucket qualifies as Hot (passes Hot gate below)
  AND abs(H4.PriceEMA20ATR) in [0.5, 1.5]   -- near H4 EMA (entry zone)
  AND H4 bias matches D1 bias                -- short-term aligned with regime
  AND H4.CI < 60                             -- H4 not actively ranging

Hot:
  D1.ADX >= 25
  AND D1.CI < 55
  AND WeeklyAlignment(W1 bias, D1 bias) != Conflict  -- Aligned or Neutral both pass;
                                                       -- only a directional W1 opposing
                                                       -- D1 blocks promotion

Watch:
  everything else
#+end_src

Demotion signals (surfaced as notes in v1, enforced as hard gates in v2):

#+begin_src text
ADX dropped below 20
CI crossed above 65
WeeklyAlignment(W1 bias, D1 bias) == Conflict   -- "W1 EMA flipped against D1 bias"
#+end_src

** Reading the output against the chart

The scanner narrows the watchlist; it does not replace looking at the
chart. This section is a guide to what each signal typically *looks like*
on a chart, since the numbers alone can be misleading without that context.

*** Bucket ≠ setup quality

A pair passing every numeric gate can still be a watch-and-wait. The
canonical failure mode: Hot or even Tradeable by the numbers, but the H4
chart shows merged/flat EMA20 and EMA50 — a consolidation, not a
trend-pullback sequence. ADX and CI describe the *daily* regime; they
cannot see that H4 has gone sideways into a coil. This is exactly what the
H4 DIST and H4 CI columns are for: use them together with a glance at the
H4 chart, not as a substitute for it.

*** EMA separation sign vs. bias

=d1.ema_sep_atr= should normally agree in sign with =bias= (positive +
long, negative + short). If they disagree — EMA20 above EMA50 but bias
reads short, for example — the pair is very likely mid-transition: the
EMA structure hasn't caught up to a fresh ±DI cross yet. This is common and
not itself alarming, but worth a chart look before trusting either signal
in isolation, especially near a Hot/Watch promotion boundary. Recall bias
here is derived from close-vs-EMA20 (=biasFromEMA=), not from ±DI, so a
transitional pair can show conflicting EMA-sep sign and bias simultaneously
without anything being "wrong" — it just means the trend is young.

*** Reading Squeeze

A Squeeze flag (H4 BB width < 2.0×H4 ATR) means price has coiled tightly
on H4. On the chart this looks like flattening, converging bands with
small-range bars. It's a *coiled-spring* signal, not a directional one —
it tells you a move is likely, not which way. Pair it with W1 alignment
before assuming continuation:

- Squeeze + W1 Aligned → coiling in the direction of the higher-timeframe
  trend; the more constructive read, worth a close look for a breakout
  continuation.
- Squeeze + W1 Conflict → ambiguous; could resolve as continuation or as
  the start of a reversal. Treat as lower conviction until it resolves.

*** Reading W1 alignment (tri-state)

- *Aligned (✓)* — W1 and D1 point the same direction. The strongest
  regime confirmation; a Hot or Tradeable pair here has multi-timeframe
  agreement.
- *Neutral (~)* — W1 is flat (price hugging the weekly EMA20). This is
  not a red flag, but it is a lower-conviction context than Aligned: the
  weekly hasn't confirmed the daily trend yet. On the chart, look for
  whether W1 is flattening because it's about to turn (watch for it
  flipping to Conflict) or just consolidating before continuing.
- *Conflict (✗)* — W1 opposes D1. The pair can still be Hot or even
  Tradeable by the numeric gates (the Hot gate only excludes hard
  conflict, not neutral), but this is the case worth the most scrutiny on
  the chart — a D1 pullback fighting the weekly structure is a
  meaningfully different risk profile than one with a clean W1 tailwind,
  even inside the same bucket.

*** TrendPct and BBPctB

=d1.trend_pct= (body/range > 0.6 over the last 20 bars) is a cleanliness
check independent of ADX — a pair can have high ADX from a few large
directional bars mixed with choppy indecision, and TrendPct exposes that.
Low TrendPct alongside high ADX is worth a chart look even in a Tradeable
bucket.

=d1.bb_pct_b= isn't currently surfaced in CLI/UI output but is computed
and available on every =ReviewResult=. Values near 0 or 1 mean price is at
a band extreme — useful context when interpreting a Squeeze or an
extended EMA-distance reading, and directly relevant groundwork for the
mean reversion classification below.

** Data requirements

| Data                        | Candles needed | Warmup driver                             |
|-----------------------------+----------------+-------------------------------------------|
| W1 EMA(20), ATR(14)         | 30 W1          | EMA(20) warmup = 20                       |
| D1 ADX(14), ATR(14), CI(14) | 60 D1          | ADX warmup = 2×14 = 28; buffer for safety |
| D1 EMA(20), EMA(50), BB(20) | 60 D1          | EMA(50) warmup = 50; covers all           |
| H4 ADX(14), ATR(14), CI(14) | 60 H4          | Same warmup ratios                        |
| H4 EMA(20)                  | 30 H4          | EMA(20) warmup = 20; 30 provides buffer   |
|-----------------------------+----------------+-------------------------------------------|


For 20 watchlist pairs that is 60 OANDA candle requests (3 per pair). A
bounded goroutine pool of 8 workers handles the fan-out without rate-limit
risk. =CandlesOptions= already supports =Count= and =Granularity=.

func (s *Service) ScanWatchlist(ctx context.Context, req scanner.ScanRequest) (*scanner.ScanResponse, error)

The service method lives in =service/scanner.go= and is a thin
fan-out: one goroutine per pair calls =scanner.ScanPair()=, which is a
pure function that takes candle slices and returns a =ScanResult=. The
service layer owns concurrency; the =scanner= package owns indicator
composition.

** TODO Mean reversion classification (strategy map)

Sequencing note: per the trading plan, this comes *after* trend-following
is comfortable and after the equities expansion — added here as a design
placeholder so the shape is ready when that time comes, not as a v1
commitment.

*** Why this doesn't fit into the existing =Bucket= field

Trend-following and mean reversion want opposite regime conditions:

| Signal      | Trend-following (current)      | Mean reversion                    |
|-------------+---------------------------------+------------------------------------|
| D1 ADX      | >= 25 (trending)                | typically < 20 (no trend)          |
| D1 CI       | < 55 (extending)                | > 65 — currently the *exclusion*   |
| Entry       | pullback toward D1 EMA20        | extension away from mean, at a     |
|             |                                  | band extreme                       |
| BB %B       | not gated on                    | near 0 or 1 (band extreme)         |
| Bias        | trend direction                 | fade direction (opposite of        |
|             |                                  | extension)                         |
| Squeeze     | secondary continuation signal   | mostly irrelevant, possibly a      |
|             |                                  | caution flag (squeezes usually     |
|             |                                  | resolve into trends, not reverts)  |

A pair can legitimately be Watch for trend and Tradeable for MR at the
same time — a ranging AUDCAD at D1 ADX 14 is exactly the case
trend-following ignores that MR would want. This means a single =Bucket
string= can't represent both classifications; the struct needs to carry
results per strategy.

*** Struct shape: strategy map

#+begin_src go
// ReviewResult gains a Strategies map alongside (or replacing) the single

type StrategyResult struct {
    Bucket string   `json:"bucket"` // "watch" | "hot" | "tradeable"
    Bias   string   `json:"bias"`   // "long" | "short" | "neutral"

This keeps =D1Snapshot=/=H4Snapshot=/=W1Snapshot= as shared raw
indicator data (computed once per pair regardless of strategy count) and
makes bucket/bias/notes strategy-scoped. Adding a third sub-strategy later
is a new map key, not a struct migration.

// ClassifyMR is the mean-reversion sibling to Classify, using inverted

#+end_src

Rough gate sketch (thresholds are starting points, not tuned):

#+begin_src text
MR Hot:
  D1.ADX < 20                      -- no established trend
  AND D1.CI > 65                   -- actively ranging

MR Tradeable:
  qualifies as MR Hot
  AND abs(D1.BBPctB - 0.5) > 0.4   -- price near a band extreme (BBPctB
                                     -- near 0 or 1)
  AND NOT setup.squeeze            -- exclude coiling pairs; a squeeze here
                                     -- more often precedes a trend than a
                                     -- reversion

MR bias:
  BBPctB > 0.9  -> "short" (fade the upper-band extension)
  BBPctB < 0.1  -> "long"  (fade the lower-band extension)
  else          -> "neutral"
#+end_src

Note =D1.BBPctB= and =D1.BBWidthATR= are already computed in
=computeD1= (pair.go) and already in the =D1Snapshot= struct — no new
candle fetches or indicators needed. This is purely a classification-layer
addition.

- CLI/UI: two bucket columns (Trend / MR), or collapse into one column
  with a strategy indicator when both fire? Two columns is simpler and
  matches the existing tabwriter layout most directly.
- Should MR Tradeable additionally require W1 to be Neutral or Conflict
  (i.e. no strong higher-timeframe trend to fight), given MR is explicitly
  a counter-trend approach at the D1 level?

** Open questions

1. *Watchlist persistence.* In v1 the watchlist defaults to the 7 majors plus
   any pairs passed as a query parameter or CLI flag. A persistent watchlist
   and hotlist implies state that survives restarts. Options: extend the global
   config YAML with a =[watchlist]= and =[hotlist]= section, or store in a
   lightweight local file. The hotlist in particular needs to survive a restart
   since it represents a deliberate daily judgment.

2. *Scan scheduling.* The endpoint is on-demand (pull) in v1. A future step
   is a scheduled daily push: run at 17:00 NY close, cache the result in an
   in-memory field on =Service= (or a JSON file in the data directory), and
   serve the cached response until the next scheduled run. Pairs the REST and
   CLI endpoints would gain a =?cached=true= flag to serve the last scan
   without re-fetching candles.

3. *S&R levels as setup signal.* When the S&R detection pipeline
   (=analysis/=) is wired into the scanner, =SetupSnapshot= gains a
   =NearSRLevel bool= field and the Tradeable gate can require either EMA
   proximity *or* S&R proximity. The struct shape is designed to accommodate
   this without breaking the API.

4. *Integration with =fx-pair-analysis-methodology.org=.* That doc specifies
   a quality scoring model (trend clarity 25%, structure clarity 25%, level
   clarity 15%, etc.) and a three-tier watchlist prioritization scheme. The
   scanner's bucket system maps directly to that model's Tradeable / Watchlist
   / No Trade labels and the =ScanPair()= function is the natural home for
   the =AnalyzePair()= logic sketched there. These should be unified rather
   than maintained separately.

5. *Multi-account.* The scan is market data, not account data, so it does not
   directly depend on multi-account support. Once multi-account is live, each
   account may maintain its own watchlist. The =ScanRequest= is parameterised
   by =Instruments= so this is solved at the call site without changing
   scanner internals.

6. *Naming drift between this doc and shipped code.* The struct/function
   section above has been updated to =ReviewResult=/=ReviewPair()=/package
   =review= based on the actual source (=review/classify.go=,
   =review/pair.go=, =review/review.go=). The Service method, REST
   endpoint, and CLI command sections below this point still use the
   original =Scan*=/=scanner= naming from the initial design and have
   *not* been cross-checked against =service/=, =api/rest/=, or =cmd/=
   source — those files weren't available for this pass. Before treating
   this doc as current, verify whether those layers were also renamed to
   =Review*= to match, or whether the package boundary genuinely differs
   between the =review= indicator/classification package and a separate
   =scanner= service-facing name.
