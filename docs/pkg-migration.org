#+title: Trader Package Migration Notes
#+author: Codex
#+date: 2026-06-23

* Goal

Break the current large root =trader= package into well-defined
packages with clear dependency direction, while keeping behavior
stable and avoiding large disruptive rewrites.

The migration should move from low-level, low-dependency concepts
outward toward orchestration code. In practice, that means shrinking
the root package from the leaves inward.

* Recommended Dependency Direction

Target shape:

#+begin_src text
cmd, api/rest, api/mcp
        |
     service
        |
 backtest / live orchestration
        |
 execution, marketdata, journal, strategy
        |
 indicators
        |
 market/domain primitives
#+end_src

The important rule is that lower-level packages should not import
higher-level packages. Domain types should not know about services,
APIs, CLI commands, OANDA clients, reports, or backtest execution.

* First Candidates

** 1. Market/domain primitives

This should be the first extraction.

Likely files:

- =types_money.go=
- =types_units.go=
- =types_tick.go=
- =types_time.go=
- =types_math.go=
- =trade_instrument.go=
- =candle_types.go=
- Later: =trade.go=, =trade_position.go=, =trade_lot.go=

Possible package names:

- =market=
- =domain=
- =fx=
- =core=

Recommendation: prefer =market= or =domain= over a generic =types= package.

Rationale:

- These are the core fixed-point and market concepts used everywhere.
- Everything else should depend on =Price=, =Money=, =Rate=, =Units=, =Pips=, =Candle=,
  =Tick=, =Timestamp=, and =Instrument=.
- Extracting this first gives later packages a stable dependency target.

Migration tactic:

Keep root-package aliases temporarily to reduce churn:

#+begin_src go
type Price = market.Price
type Money = market.Money
type Candle = market.Candle
#+end_src

Then migrate callers package by package.

** 2. Indicators

This is the cleanest early extraction after primitives.

Likely files:

- =indicators.go=
- =indicators_fixed.go=
- =indicators_ema.go=
- =indicators_atr.go=
- =indicators_adx.go=
- =indicators_bollinger.go=
- =indicators_choppiness.go=

Target package: =indicators=

Rationale:

- Indicators are pure domain logic.
- They should depend only on fixed-point/domain types.
- They should not know about =Backtest=, =Broker=, =Account=, config, service, files, OANDA,
  or strategy registration.

This is a high-value split because it creates a small, testable package with a clean API.

** 3. Journal persistence

Likely files:

- =journal.go=
- =journal_csv.go=
- =journal_json.go=
- =journal_org.go=
- Maybe keep =journal_live.go= separate initially because it has live/OANDA concerns.

Target package: =journal=

Rationale:

- The =Journal= interface is already a package boundary.
- CSV, JSONL, and Org writers are persistence adapters.
- This package should depend on domain trade/equity types, not on the backtest engine.

Suggested rule:

Keep live journal streaming out of the first pass unless its dependencies are already clean.

** 4. Candle store and market data

Likely files:

- =store.go=
- =store_inventory.go=
- =store_key.go=
- =store_keymap.go=
- =data_manager.go=
- =data_plan.go=
- =data_wants.go=
- =data_writer.go=
- =data_validation.go=
- =candle_iter.go=
- =candle_agg.go=
- =candle_gap.go=
- =candle_set.go=

Possible package names:

- =marketdata=
- =candles=
- =store=

Recommendation: =marketdata= is probably the clearest if the package owns loading, planning,
validation, and candle iteration. =store= is narrower and may fit only the filesystem/cache
pieces.

Rationale:

- The data manager and store code are a natural subsystem.
- This split should come after the domain primitive extraction because these files depend heavily
  on =Candle=, =Instrument=, =Timestamp=, and =Price=.

** 5. Accounting and execution

Likely files:

- =account.go=
- =account_close.go=
- =account_sizing.go=
- =broker.go=
- =broker_event.go=
- =trade_request.go=
- =trade_history.go=
- Possibly =trade.go=, =trade_position.go=, =trade_lot.go= if not already moved to domain.

Possible package names:

- =accounting=
- =execution=
- =broker=
- =portfolio=

Rationale:

- This area owns critical invariants: equity, margin, P/L, spread semantics, open/close behavior,
  and order requests.
- It is an important boundary, but it is more delicate than indicators or journals.

Recommendation:

Do not make this the first split. Move it after fixed-point/domain primitives are stable and
after the pure packages are out of the root package.

** 6. Backtest engine, strategies, exits, and regimes

Likely files:

- =backtest*.go=
- =trader.go=
- =strategy*.go=
- =regime_filter*.go=

Possible package names:

- =backtest=
- =engine=
- =strategy=

Rationale:

- This is the orchestration layer and currently ties together many concepts.
- Strategy interfaces depend on =Backtest=, =StrategyPlan=, candle timing, exits, regimes, and
  registration.
- Splitting this too early risks just moving the current knot into multiple directories.

Recommendation:

Handle this after the lower-level packages exist. The extraction will be much easier once
market primitives, indicators, journals, data, and execution/accounting already have clear APIs.

* Suggested Migration Order

1. Extract =market= or =domain= primitives.
2. Extract =indicators=.
3. Extract =journal=.
4. Extract =marketdata= / =candles= / =store=.
5. Extract =accounting= and/or =execution=.
6. Extract =backtest= / =engine= / =strategy= boundaries.

* Practical Migration Rules

- Keep behavior unchanged during package moves.
- Use type aliases in the old root package temporarily to keep churn manageable.
- Move tests with their packages when possible.
- Prefer one package boundary per PR or commit.
- Run =gofmt= after each move.
- Run focused package tests first, then =make test=.
- Preserve the no-floats internal calculation rule throughout the migration.
- Avoid putting APIs, CLI handlers, or OANDA-specific wire concerns into domain packages.

* Notes

Do not start by splitting =service/=. It already acts as an application boundary between the
transport layers and the core engine. The larger problem is that the root =trader= package
currently contains domain types, indicators, data storage, accounting, journals, strategy
registration, and backtest orchestration.

The best path is to reduce the root package gradually from stable low-level concepts outward.
